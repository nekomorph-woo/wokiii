# 设计文档系统性审阅

对设计文档目录执行实体级索引 + 多维交叉审阅 + 歧义拷问，输出结构化审阅报告。核心价值：在人工审阅前先把文档对齐到可审阅状态。

**工作流程执行过程中，务必遵守【核心约束】和【检查清单】中的强制硬性要求，流程执行完最后步骤，遵照【检查清单】的每项内容进行仔细检查，发现问题时优先反思处理，无法解决时再使用 AskUserQuestion 工具向用户确认或求助，每项内容检查无误时将【检查清单】的检查结果追加到控制台界面。**

## 输入

使用 AskUserQuestion 收集：

1. **设计文档目录**：包含设计文档的目录路径
2. **输出目录**：审阅报告输出路径（若未指定则询问）
3. **审阅维度**（可选）：默认全部 5 维，用户可跳过或添加自定义维度

## 工作流程

```
索引构建 → 确认索引 → 多维审阅 → 歧义拷问 → 反向修补文档 → 输出报告
```

### Phase 1：实体级索引构建

读取全部设计文档，提取 5 类实体（详见 [reference/indexing-guide.md](reference/indexing-guide.md)）：

- **类型实体**：类/接口/枚举的字段、方法签名
- **关系实体**：依赖/引用/实现/继承的方向
- **契约实体**：API 入参出参、前置条件
- **场景实体**：用例/分支路径/异常路径
- **约束实体**：设计原则/架构规则/不变量

输出 `00-review-index.md`，**暂停等待用户确认索引准确性**后再进入 Phase 2。

### Phase 2：多维审阅

按配置的维度执行审阅（详见 [reference/review-dimensions.md](reference/review-dimensions.md)）：

| 维度 | 名称 | 核心检查 |
|------|------|----------|
| D1 | 纵向追溯 | PRD需求→架构决策→模块设计 是否逐层覆盖 |
| D2 | 横向一致 | 同一实体跨文档定义是否矛盾 |
| D3 | 完整性 | 引用悬空、场景遗漏、依赖未说明 |
| D4 | 依赖验证 | 方向合理性、循环依赖、外部依赖明确度 |
| D5 | 假设暴露 | 隐含假设、未验证的外部行为、原则违反 |

每个发现标记严重度：🔴 阻塞 / 🟡 警告 / 🟢 建议 / ❓ 待确认。

### Phase 3：歧义深拷问

对 Phase 2 标记为 ❓ 的发现，使用 /grill-me 逐个拷问用户：

1. 呈现发现 + 影响范围
2. 拷问决策树，逼出明确结论
3. 记录决策 + 连锁影响
4. 连锁更新受影响的其他发现

### Phase 3.5：反向修补设计文档

根据 Phase 2 的发现和 Phase 3 的决策，反向修改原始设计文档：

**自动修复（无需确认）**：字段名/类型不一致、引用悬空、章节号失效、拷问后有明确结论的项

**需确认后修复**：🔴 级别的结构性问题（涉及设计意图变更）→ 展示 diff 让用户确认后再修改；涉及多文档连锁修改 → 展示影响范围后批量修改

**不修改**：🟡🟢 建议级发现和未获得决策的 ❓ 项 → 仅记入报告

### Phase 4：审阅报告

输出 7 份报告，编号遵循 [reference/report-numbering.md](reference/report-numbering.md)：
`00-review-index`（模板见 [indexing-guide.md §3](reference/indexing-guide.md)） → `01-traceability` → `02-consistency` → `03-completeness` → `04-assumptions`（模板见 [report-dimension-templates.md](reference/report-dimension-templates.md)） → `05-decisions` → `06-action-items`（模板见 [report-summary-templates.md](reference/report-summary-templates.md)）

`06-action-items` 末尾包含后续步骤提示：待处理项由用户读报告后用 /grill-me 决策或自行决策，反向修改文档，完成后重跑本技能验证。

## 核心约束

- **索引确认点**：Phase 1 完成后必须暂停让用户确认，索引不准则审阅全偏
- **不替用户做决定**：❓ 待确认项只呈现冲突和影响，不替用户选方向
- **连锁影响追踪**：用户决策后立即检查对其他发现的影响，更新受影响项
- **修补再报告**：Phase 3.5 先修补文档，Phase 4 的报告反映修补后状态
- **分级修补**：机械性问题自动修、结构性问题确认后修、建议级不动
- **禁止自动 commit**：等用户指令

## 检查清单

工作流程最后一步完成后，逐项检查并将结果输出到控制台。详见 [reference/checklist.md](reference/checklist.md)。
