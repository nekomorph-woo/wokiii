# 审阅维度详细说明

## D1 纵向追溯 (Traceability)

### 目标

验证需求→架构→模块设计的逐层覆盖，发现"有需求无设计"或"有设计无需求"的缺口。

### 检查项

#### 需求→架构覆盖

- PRD 中的每个功能点是否有对应的架构决策？
- PRD 中的非功能需求（性能、安全、可用性）是否在架构中体现？
- PRD 中的边界条件是否在架构中有处理策略？

#### 架构→模块设计覆盖

- 架构中定义的每个模块是否有对应的子模块设计文档？
- 架构中定义的接口/契约是否在子模块中展开？
- 架构中的全局约束是否在每个子模块中遵守？

#### 跨层一致性

- 子模块中引用的概念是否在架构层有定义？
- 子模块的设计是否与架构决策方向一致？
- 是否存在子模块设计偏离架构原则的情况？

### 输出格式

```markdown
## 追溯矩阵

| 需求/决策 | 架构覆盖 | 模块设计覆盖 | 状态 |
|-----------|----------|-------------|------|
| PRD:R01   | 02:§3.1  | 03b:§3.5    | ✅   |
| PRD:R12   | —        | —           | 🔴 无设计 |
```

## D2 横向一致 (Consistency)

### 目标

验证同一实体在不同文档中的定义是否一致，发现字段名/类型/语义冲突。

### 检查项

#### 类型一致性

- 同名类/接口在不同文档中的字段列表是否一致？
- 字段类型（含泛型参数）是否匹配？
- 方法签名（参数+返回值）是否匹配？
- 枚举值集合是否一致？

#### 语义一致性

- 同一概念在不同文档中的描述是否语义相同？
- 同一操作的命名是否统一（如 "过滤" vs "筛选" vs "过滤"）？
- 场景码/状态码的语义是否在所有引用处一致？

#### 引用完整性

- 文档 A 引用文档 B 中定义的实体时，名称是否精确匹配？
- "详见 XX" 的引用目标是否存在？
- 引用的章节号是否仍然有效？

### 输出格式

```markdown
## 不一致发现

### 🔴/🟡/🟢 #N

| 属性 | 位置A | 位置B |
|------|-------|-------|
| 字段类型 | `Map<String,String>` (02:§4.1) | `Map<String,Object>` (03d:§6) |

**差异说明**：...
**建议**：...
```

## D3 完整性 (Completeness)

### 目标

发现引用悬空、场景遗漏、依赖未说明等缺口。

### 检查项

#### 引用完整性

- 所有跨文档引用的实体是否有对应定义？（悬空引用检测）
- 所有"详见 XX"的引用目标是否存在？
- 代码示例中引用的类型是否在索引中有定义？

#### 场景完整性

- 列举的所有场景是否有对应的处理流程？
- 分支路径是否都有 fallback/降级设计？
- 异常路径是否都有错误处理？
- 边界条件（空值、上限、并发）是否覆盖？

#### 依赖完整性

- 声明依赖的外部系统是否有接口说明？
- 依赖的数据结构是否有来源说明？
- 隐式依赖（如配置项、环境变量）是否声明？

### 输出格式

```markdown
## 缺口清单

### 🔴/🟡/🟢 #N

**类型**：悬空引用 / 场景遗漏 / 依赖未说明
**位置**：...
**缺口内容**：...
**影响范围**：...
**建议**：...
```

## D4 依赖验证 (Dependency)

### 目标

验证依赖方向合理性、外部依赖明确度、共享/独属划分。

### 检查项

#### 方向性

- 依赖方向是否符合分层规则？（如 domain 不依赖 infrastructure）
- 是否存在循环依赖？
- 依赖方向是否与文档声明的架构原则一致？

#### 外部依赖

- 外部系统接口是否明确了请求/响应格式？
- 外部系统的 SLA/可用性假设是否声明？
- 外部系统不可用时的降级策略是否设计？

#### 划分合理性

- 共享模块 vs 模块私有的边界是否一致？
- ACL 的读写分离是否清晰？
- 放在 shared_model 的类型是否确实被多层使用？
- 放在模块私有的类型是否确实只有一个消费者？

### 输出格式

```markdown
## 依赖问题

### 🔴/🟡/🟢 #N

**类型**：方向违反 / 循环依赖 / 外部依赖不明确 / 划分不合理
**涉及**：...
**问题说明**：...
**建议**：...
```

> **编号规则**：D3 完整性缺口和 D4 依赖问题在同一文档（03-completeness.md）中，编号从 #1 连续递增，不分子类别重置。如缺口编到 #10，依赖问题从 #11 开始。

## D5 假设暴露 (Assumptions)

### 目标

暴露文档中隐含但未声明的假设，检测被违反的设计原则。

### 检查项

#### 隐含假设

- 设计中隐含了"某个外部系统会提供 X"但未显式声明
- 设计中假设了"数据格式是 Y"但未与数据源确认
- 设计中默认"某个场景不会发生"但无排除说明
- 设计中假设了"某个操作是幂等的"但未声明
- 设计中假设了"某个接口的响应时间/数据量"但未验证

#### 原则违反

- 文档中声明的架构原则是否被自己的设计违反？
- 约束文档中的规则是否在子模块设计中被绕过？
- 设计约束之间是否自相矛盾？

#### 遗漏的替代方案

- 关键设计决策是否考虑了替代方案？
- 选择当前方案的理由是否记录？
- 是否存在未讨论的更简单方案？

### 输出格式

```markdown
## 假设/原则发现

### 🔴/🟡/🟢/❓ #N

**类型**：隐含假设 / 原则违反 / 替代方案遗漏
**位置**：...
**发现内容**：...
**如果假设不成立的影响**：...
**建议**：声明假设 / 验证假设 / 调整设计
```

## 自定义维度

用户可添加自定义审阅维度，格式：

```
维度名：简短名称
检查重点：该维度关注什么
检查项：
  - 具体检查项 1
  - 具体检查项 2
```

自定义维度与内置维度享有同等严重度分级和输出格式。
