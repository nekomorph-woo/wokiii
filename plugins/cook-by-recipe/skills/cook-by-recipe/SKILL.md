# 测试驱动开发

## 核心原则

**测试验证行为，而非实现细节。** 代码可以完全改变，测试不应受影响。

**GOOD tests**通过公共接口验证真实代码路径，描述系统做什么，而非怎么做。测试如规格说明书——"用户可用有效购物车结账"直接说明存在什么能力。此类测试经受内部重构，因为不关心内部结构。

**BAD tests**与实现耦合。它们 mock 内部协作者、测试私有方法、或通过外部手段验证（如直接查数据库而非使用接口）。警告信号：重构时代码行为未变但测试失败。若重命名内部函数导致测试失败，说明测试的是实现而非行为。

详见 [tests.md](reference/tests.md) 和 [mocking.md](reference/mocking.md)。

## 适用范围

**判断标准**：代码是否包含需要验证的业务规则或复杂逻辑。

| 判断 | 说明 |
|------|------|
| 需要 TDD | 有复杂逻辑、业务规则、边界条件需要验证 |
| 不需要 TDD | 只是传递数据、配置框架、调用第三方 |

**需要 TDD 的代码示例**：

- 核心业务规则（价格计算、折扣逻辑、状态转换）
- 复杂条件分支（多条件组合的业务判断）
- 边界条件处理（金额上限、时间边界）
- 数据转换/映射逻辑
- 错误处理逻辑
- 并发/异步逻辑
- 安全相关代码（认证、授权、加密）
- 数据验证/校验逻辑

**不需要 TDD 的代码示例**：

- 简单数据传递（getter/setter、DTO）
- 框架胶水代码（依赖注入配置、路由配置）
- 第三方 SDK 调用（集成点，用集成测试更合适）
- 日志记录（辅助功能）
- 简单 CRUD（基础数据存取）
- 样板代码（框架要求的固定结构）
- 缓存层（行为透明）
- 中间件配置

**DO NOT** 询问用户哪些代码需要 TDD。若用户未明确指定，基于上述标准自行判断。若用户需要指定特定代码使用 TDD，由用户主动说明。

## 反模式：水平切片

**DO NOT** 先写完所有测试，再写所有实现。这是"水平切片"——把 RED 当作"写所有测试"，GREEN 当作"写所有代码"。

这产生**垃圾测试**：

- 批量测试验证的是想象的行为，而非实际行为
- 最终测试的是事物的形状（数据结构、函数签名），而非用户面对的行为
- 测试对真实变化不敏感——行为损坏时通过，行为正常时失败
- 在理解实现之前就承诺测试结构，超出视野范围

**正确做法**：通过 tracer bullet 进行垂直切片。一个测试 → 一个实现 → 重复。每个测试响应上一轮学到的内容。因为刚写完代码，确切知道什么行为重要以及如何验证。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流程

### 1. 规划

编写代码前，检查上游是否有执行计划：

**读取上游**（可选）：检查 `plans/<feature-name>/_plan.md` 是否存在：
- **存在**：读取 frontmatter 和执行步骤，作为 TDD 循环的输入。执行步骤中的验证标准直接映射为测试断言目标
- **不存在**：从当前对话上下文和代码库中获取必要信息

然后确认：

- [ ] 与用户确认需要的接口变更
- [ ] 确认要测试的行为（优先级排序）
- [ ] 列出要测试的行为（非实现步骤）
- [ ] 获取用户确认

询问："公共接口应该是什么样？哪些行为最重要需要测试？"

**无法测试所有内容。** 与用户确认哪些行为最关键。聚焦关键路径和复杂逻辑，而非所有边界情况。

### 2. Tracer Bullet

编写一个测试，验证系统的一件事：

```
RED:   编写第一个行为的测试 → 测试失败
GREEN: 编写使测试通过的最小代码 → 测试通过
```

这是 tracer bullet——端到端证明路径可行。

### 3. 增量循环

对每个剩余行为：

```
RED:   编写下一个测试 → 失败
GREEN: 使测试通过的最小代码 → 通过
```

规则：

- 一次一个测试
- 只写使当前测试通过的最小代码
- 不要预判未来测试
- 保持测试聚焦于可观察的行为

### 4. 重构

所有测试通过后，寻找[重构候选](reference/refactoring.md)：

- [ ] 提取重复
- [ ] 深化模块（将复杂度移到简单接口之后）
- [ ] 在自然处应用 SOLID 原则
- [ ] 思考新代码揭示的现有代码问题
- [ ] 每步重构后运行测试

**RED 状态下不要重构。** 先回到 GREEN。

## 每轮检查清单

```
[ ] 测试描述行为，而非实现
[ ] 测试仅使用公共接口
[ ] 测试能经受内部重构
[ ] 代码对当前测试是最小的
[ ] 没有添加投机性功能
```
