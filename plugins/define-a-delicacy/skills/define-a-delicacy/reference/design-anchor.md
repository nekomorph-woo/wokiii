# 设计锚点

设计锚点是 What 向 How 的衔接点。它收窄下游设计空间，但不指定具体方案。

## 本质

| 维度 | 设计锚点 | 接口（属于 How 层） |
|------|----------|---------------------|
| 精度 | 方向约束 | 确定约束 |
| 读者 | 设计师（人或 AI） | 开发者（人或 AI） |
| 作用 | 排除不合理方案 | 可直接编码 |
| 示例 | "认证必须在一个请求内完成" | `authenticate(token: string): UserPayload` |

## 四类锚点

### 1. 效果锚点

约束用户体验或系统行为的效果：

- "登录必须在一次 HTTP 往返内完成"
- "支付回调必须在 3 秒内响应"
- "搜索结果必须在 200ms 内返回"

### 2. 安全锚点

约束安全边界：

- "密码不得明文存储"
- "登录失败不暴露用户是否存在"
- "权限检查不能依赖数据库查询"

### 3. 必要性锚点

标记必须存在的功能：

- "支持后续扩展多因素认证"
- "必须支持离线模式"
- "必须向后兼容 v1 API"

### 4. 排除锚点

标记明确不做的事：

- "不引入第三方认证服务"
- "不支持实时推送（本次版本）"
- "不做多租户隔离"

## 写法规则

- 每条锚点一句话，不超过 30 字
- 使用"必须 / 不得 / 不 / 不支持"等明确措辞
- 数量：3-8 条
- **DO NOT** 包含技术实现细节（函数名、类名、数据结构）
