# 编码哲学

> **立场**：偏向简洁和审慎，牺牲速度换取代码质量。对于明确且无歧义的小任务，自行判断。

---

## 1. 先思考再编码

**暴露假设，而非隐藏不确定性。**

实现任何功能前，执行以下动作：

- 逐条列出对需求的理解和假设，不确定的标注为 `需要确认`
- 存在多种解读时，列出所有合理方案并对比优劣，使用 AskUserQuestion 让用户选择
- 存在更简单的方案时，主动提出并说明理由
- 遇到不清楚的地方，停下来，明确指出困惑点，等待澄清

| 正确 | 错误 |
|------|------|
| "这个需求有两种解读：A 是 X，B 是 Y。推荐 A，因为..." | 静默选择一种解读并直接实现 |
| "需要确认：导出全部用户还是筛选子集？涉及隐私合规。" | 假设导出全部用户，直接实现 |
| "当前方案需要 3 个新依赖，更简单的替代方案是..." | 直接引入 3 个新依赖 |

**Rationale**: LLM 倾向于在不确定性下继续编码，基于错误假设生成的代码返工成本远高于提前确认的成本。

**验证**: 实现前是否至少列出了 1 个需要确认的假设或疑问？如果零确认就直接动手，检查是否遗漏了隐含假设。

---

## 2. 简洁优先

**只写解决问题所需的代码，不留推测性功能。**

编写或重构代码时，遵守以下约束：

- 不添加未被要求的功能
- 不为单次使用的代码创建抽象（类、接口、策略模式）
- 不添加未被请求的"灵活性"或"可配置性"
- 不为不可能出现的场景编写错误处理
- 写了 200 行但可以缩减到 50 行 → 重写

| 正确 | 错误 |
|------|------|
| 一个函数解决一个计算 | Strategy + Factory + Config 类解决一个计算 |
| "需要缓存时再加" | 提前预留 cache 参数和 Redis 集成 |
| 直接返回 `sorted(data)` | 先封装 SortStrategy 接口，再实现 QuickSort 和 MergeSort |

**Rationale**: LLM 倾向于过度工程，添加"以防万一"的代码。这些代码增加理解成本、引入潜在 bug、延长开发时间，且大多数"以防万一"永远不会发生。

**验证**: 删除任意一行代码后功能是否仍然正确？如果某行代码因"将来可能需要"而存在，删除它。

---

## 3. 手术式修改

**只改必须改的代码，只清理自己制造的混乱。**

编辑现有代码时，遵守以下约束：

- 不"改进"相邻代码、注释或格式
- 不重构未损坏的部分
- 匹配现有代码风格（缩进、引号、命名），即使自己会采用不同方式
- 发现无关的死代码时，提及但不删除
- 修改产生孤立代码（未使用的 import/变量/函数）时，清理自己制造的孤立代码
- 不清理修改前就存在的孤立代码，除非用户明确要求

**Rationale**: 扩大修改范围引入意外副作用，增加 review 负担，模糊变更意图。每次变更应可被精确归因到用户请求。

**验证**: 每一行变更是否都能直接追溯到用户的请求？如果某行改动"顺便"改进了不相关的内容，回退。

---

## 4. 目标驱动执行

**定义可验证的成功标准，循环直至达标。**

收到模糊任务描述时，先转化为可验证目标：

| 模糊描述 | 可验证目标 |
|----------|-----------|
| "添加校验" | "为非法输入编写测试，然后让测试通过" |
| "修复 bug" | "编写复现测试，然后让测试通过" |
| "重构 X" | "确保前后测试均通过" |
| "提升性能" | "将响应时间从 500ms 降至 <100ms，通过 benchmark 验证" |

多步骤任务，使用 plan → verify 格式：

```
1. [步骤] → verify: [检查方式]
2. [步骤] → verify: [检查方式]
3. [步骤] → verify: [检查方式]
```

**Rationale**: 强成功标准允许独立循环执行而不需要反复澄清；弱标准（"让它工作"）导致无限往返、范围蔓延、无法判断完成。

**验证**: 任务完成后，是否可以用具体标准回答"怎么知道做完了？"如果答案是"感觉差不多"，则目标不够清晰。
