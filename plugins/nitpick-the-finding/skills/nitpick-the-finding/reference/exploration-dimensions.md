# 探索维度详细规范

## 1. 架构探索

**目标**：理解代码的宏观结构。

**探索方式**：

- 目录结构分析（`find` + `ls`）
- 入口文件识别（main、index、app 等）
- 模块/包的划分和依赖关系
- 分层结构（controller / service / repository 等）

**输出**：模块关系图（ASCII） + 分层描述

## 2. 模式探索

**目标**：识别代码中已建立的设计约定。

**探索方式**：

- 常见设计模式的使用（factory、strategy、observer 等）
- 命名约定（文件名、函数名、变量名）
- 错误处理模式
- 配置管理方式
- 测试组织方式

**输出**：模式列表 + 每个模式的出现位置和用法

## 3. 约束探索

**目标**：识别不可逾越的技术约束。

**探索方式**：

- 包管理文件（package.json、requirements.txt、go.mod 等）
- 框架版本和配置
- 外部服务依赖（数据库、消息队列、第三方 API）
- 环境变量和配置文件
- 编码规范文件（.eslintrc、pyproject.toml 等）

**输出**：约束列表 + 影响评估

## 4. 痛点探索

**目标**：识别代码中的脆弱区域。

**探索方式**：

- 大文件（超过 300 行）
- 高耦合（同一文件被大量其他文件引用）
- 重复代码（相似逻辑出现在多处）
- TODO/FIXME/HACK 注释
- 缺少测试的核心模块

**输出**：痛点列表 + 严重度 + 影响范围

## 5. 术语探索

**目标**：理解领域概念在代码中的映射。

**探索方式**：

- 关键业务词汇在代码中的出现位置
- 同一概念是否有多种命名
- 外部术语（API 响应字段、数据库列名）和内部术语的映射

**输出**：术语表 + 歧义标注

## 6. 代码→文档映射

**目标**：建立代码与设计文档的对应关系，识别无文档覆盖的代码区域。

**映射方式**：

4 层反向搜索：

| 层级 | 搜索方式 | 匹配精度 |
|------|----------|----------|
| 类名精确匹配 | 代码中的类名 → `plans/` 下 design.md 的模块名 | 高 |
| 方法名匹配 | 代码中的函数名 → design.md 接口契约 | 中 |
| 业务概念 | 代码中的领域词汇 → _define.md 用户故事和设计锚点 | 中 |
| 模块映射 | 代码目录结构 → _registry.md 模块列表 | 低 |

**输出**：映射表（代码区域 → 设计文档路径 → 状态：✅ 已有 / ⚠️ 无文档）

**用途**：标记 ⚠️ 的区域是后续设计的优先覆盖范围。

## 并行执行

使用 Agent 子代理并行探索各维度，每个维度一个子代理。探索完成后在主上下文中综合输出。

不要在主上下文中逐个文件读取 — 使用子代理保护主上下文窗口。
