# 设计接口

基于 "Design It Twice" 原则（《软件设计哲学》）：第一个方案通常不是最优解。生成多个差异显著的设计，再对比选择。

## 工作流程

### 1. 收集需求

设计前明确：

- [ ] 模块解决什么问题？
- [ ] 调用方是谁？（其他模块、外部用户、测试）
- [ ] 关键操作有哪些？
- [ ] 有何约束？（性能、兼容性、现有模式）
- [ ] 什么应该隐藏？什么应该暴露？

询问："模块需要做什么？谁会使用它？"

### 2. 生成设计（并行 Sub-Agent）

使用 Agent 工具同时启动 3+ 个 sub-agent。每个必须产出**差异显著**的方案。

```
每个 sub-agent 的提示模板：

为以下模块设计接口：[模块描述]

需求：[收集的需求]

本设计的约束：[为每个 agent 分配不同约束]
- Agent 1: "最小化方法数量 - 最多 1-3 个方法"
- Agent 2: "最大化灵活性 - 支持多种用例"
- Agent 3: "优化最常见场景"
- Agent 4: "参考 [特定范式/库] 的设计风格"

输出格式：
1. 接口签名（类型/方法）
2. 使用示例（调用方如何使用）
3. 内部隐藏了什么
4. 该方案的权衡
```

### 3. 展示设计

每个设计展示：

1. **接口签名** - 类型、方法、参数
2. **使用示例** - 调用方的实际用法
3. **隐藏内容** - 内部保持的复杂度

依次展示，确保用户理解每个方案后再对比。

### 4. 对比设计

展示全部设计后，从以下维度对比：

- **接口简洁性**：方法更少、参数更简单
- **通用 vs 专用**：灵活性 vs 聚焦性
- **实现效率**：接口形态是否允许高效的内部实现
- **深度**：小接口隐藏大量复杂度（好）vs 大接口仅有薄实现（坏）
- **正确使用难度** vs **误用难度**

用文字讨论权衡，而非表格。突出设计间分歧最大的地方。

### 5. 综合

最优设计往往综合多个方案的洞察。询问：

- "哪个设计最符合你的主要用例？"
- "其他设计中有值得吸收的元素吗？"

## 评估标准

来自《软件设计哲学》：

**接口简洁性**：方法更少、参数更简单 = 更易学习和正确使用。

**通用性**：无需修改即可支持未来用例。但避免过度泛化。

**实现效率**：接口形态是否允许高效实现？还是迫使内部实现变得别扭？

**深度**：小接口隐藏大量复杂度 = 深度模块（好）。大接口仅有薄实现 = 浅模块（避免）。

## 反模式

- **DO NOT** 让 sub-agent 产出相似设计 - 强制差异显著
- **DO NOT** 跳过对比 - 价值在于差异对比
- **DO NOT** 开始实现 - 此阶段仅关注接口形态
- **DO NOT** 基于实现难度评估 - 评估接口本身
