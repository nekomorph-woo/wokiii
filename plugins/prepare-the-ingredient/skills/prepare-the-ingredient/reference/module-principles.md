# 模块设计原则

## Deep Module

来自 *A Philosophy of Software Design*（John Ousterhout）。

**目标**：接口简单，实现复杂度隐藏在内部。

| 指标 | Deep Module | Shallow Module |
|------|------------|----------------|
| 接口复杂度 | 低 | 高 |
| 实现复杂度 | 高 | 低 |
| 用户心智负担 | 小 | 大 |
| 设计质量 | 好 | 差 |

**判断方法**：

- 接口的参数个数（少 = 好）
- 用户需要理解的 concepts 数量（少 = 好）
- 接口是否暴露了实现细节（不暴露 = 好）

**在模块识别阶段**：如果一个"模块"的接口比实现还复杂，考虑合并到其他模块。

## 自闭环模块（垂直切片）

每个模块应该独立可测试、可交付。

**特征**：

- 一个模块对应一个可验证的用户价值（tracer bullet）
- 模块内部包含从接口到数据到逻辑的完整垂直切片
- 不依赖其他模块的内部实现（只依赖公开接口）

**检验标准**：

- 能否为这个模块单独写集成测试？
- 能否单独部署这个模块的功能？
- 移除这个模块后，系统是否仍能编译运行（可能功能缺失）？

**反面模式**：

- 按技术层拆分（Phase 1: 数据库 → Phase 2: 后端 → Phase 3: 前端）
- 按角色拆分（模块 A 只做 CRUD → 模块 B 只做业务逻辑 → 模块 C 只做 UI）

这些是瀑布开发穿上 AI 外套。

## 依赖方向

**规则**：依赖方向单向，无循环。

```
✅ 合理                    ❌ 循环依赖
  A → B → C                  A → B
                              ↑   ↓
                              D ← C
```

**分层约束**：

- 核心模块（领域逻辑）不依赖外围模块（基础设施、UI）
- 外围模块可以依赖核心模块
- 工具模块（utils）不依赖任何业务模块

**检测**：构建依赖图（ASCII），检查是否有环。发现环时暂停并报告。
