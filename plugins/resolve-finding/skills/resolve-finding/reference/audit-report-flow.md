# 审阅报告模式详细流程

> 本文件为 resolve-finding skill 的审阅报告模式提供详细指导。
> 仅在模式检测为"审阅报告模式"时由 Claude 读取。

## 1. 读取审阅报告

### 1.1 报告文件定位

输入通常指向 `*action-items*`、`*review*`、`*finding*`、`*audit*` 等审阅报告文件。读取报告，定位到用户指定的待处理项。

### 1.2 提取发现信息

从报告项中提取：

| 信息维度 | 说明 | 用途 |
|----------|------|------|
| 严重度 | 🔴高 / 🟡中 / 🟢低 | 决定 grill-me 追问深度 |
| 审阅维度 | 追溯/一致性/完整性/依赖/假设 | 确定分析角度 |
| 问题描述 | 发现的具体问题 | 构建 grill-me 上下文 |
| 建议操作 | 报告给出的修复建议 | 作为推荐方案参考 |
| 涉及文档 | 报告引用的设计文档位置 | 定位需修补的文件 |

### 1.3 确认处理范围

将提取的信息呈现给用户确认：

```
📋 待处理审阅报告项：

- **严重度**: 🟡中
- **维度**: 一致性
- **问题**: <问题描述>
- **建议**: <建议操作>
- **涉及文档**: <文档路径列表>

是否处理此项？是否需要调整范围？
```

---

## 2. 定位设计文档

### 2.1 读取引用位置

报告项中引用的设计文档位置可能有多种格式：
- `03a-core-models.md §2.3` → 直接定位到文件和章节
- `03b-device-proxy-workflow.md` → 文件级引用，需通读定位相关段落
- `src/.../AgentDeviceServiceClient.java` → 代码引用，需反向确认设计文档

### 2.2 读取设计文档

- Read 设计文档中引用的章节
- 若引用为文件级 → 通读后定位与发现相关的段落
- 若引用了代码位置 → 同时 Read 代码文件，对比代码与设计文档

### 2.3 交叉验证

对照报告项的描述与设计文档实际内容，确认：
- 报告描述的问题是否仍然存在（可能已被其他流程修复）
- 问题的影响范围是否准确
- 是否遗漏了其他受影响的位置

---

## 3. /grill-me 上下文构建

审阅报告模式中，/grill-me 的输入上下文应包含：

### 3.1 必呈内容

1. **报告项原文**：严重度、维度、问题描述、建议操作
2. **设计文档现状**：相关章节原文
3. **代码现状**（若涉及代码引用）：实际代码片段
4. **交叉验证结果**：报告描述 vs 实际现状的差异

### 3.2 追问重点

- 是否认同报告中的问题定性？（严重度是否需要调整？）
- 建议操作是否合适？是否有更好的方案？
- 修复方案是否影响其他文档章节？
- 是否存在报告未覆盖的关联问题？

---

## 4. 反向修补

按 /grill-me 决策修改设计文档。通用步骤见 [common-steps.md](common-steps.md#反向修补)。

审阅报告模式默认输出到设计文档（无需用户选择输出目标）。

---

## 5. 连锁检查

### 5.1 扫描报告中其他待处理项

检查本次决策是否连带解决了报告中的其他待处理项：

```
📋 连锁检查结果：

| # | 其他待处理项 | 是否被连带解决 | 说明 |
|---|------------|--------------|------|
| 3 | 03a §1.2 参数定义不一致 | ✅ 已解决 | 本次修改统一了参数定义 |
| 7 | 05 §2.1 缺少降级说明 | ❌ 无关 | 涉及不同模块 |

是否将 #3 一并标记为已解决？
```

### 5.2 跨文档影响检查

通用连锁检查逻辑见 [common-steps.md](common-steps.md#连锁检查)。

### 5.3 连锁深度控制

通用深度控制规则见 [common-steps.md](common-steps.md#连锁深度控制)。

## 6. 状态更新

### 6.1 更新审阅报告

在报告文件中更新该项状态：

```markdown
- [x] ~~问题描述~~ → ✅ 已解决
  - **决策**: <一句话决策结论>
  - **修改文件**: <设计文档路径列表>
  - **解决时间**: <YYYY-MM-DD>
```

### 6.2 记录决策内容

在状态更新中简要记录：
- 决策结论
- 修改的设计文档
- 关联解决的报告项（如有）
