# 代码发现模式详细流程

> 本文件为 resolve-finding skill 的代码发现模式提供详细指导。
> 仅在模式检测为"代码发现模式"时由 Claude 读取。

## 1. 代码分析

当输入指向代码文件或代码级描述时：

### 1.1 定位代码文件

- 输入包含具体文件路径 → 直接 Read
- 输入为类名/方法名描述（无路径）→ Grep 在 `src/` 下搜索定位文件
- 输入为自由描述的代码级问题 → 先提取关键标识符，再 Grep 定位

### 1.2 识别代码结构

从代码文件中提取：

| 分析维度 | 提取内容 | 用途 |
|----------|---------|------|
| 包路径 | `package com.imou.agent3.xxx` | 映射到模块和设计文档 |
| 类结构 | 类名、extends/implements、字段 | 理解设计意图 |
| 公共方法 | 方法签名、参数类型、返回值 | 与设计文档对比 |
| 关键依赖 | import 中外部系统引用 | 确认 ACL 边界 |
| 所属模块 | 根据包路径推断 | 缩小设计文档搜索范围 |

### 1.3 分类发现类型

| 发现类型 | 说明 | 示例 |
|----------|------|------|
| **遗漏** | 设计文档要求的能力在代码中缺失 | 设计文档规定支持 deviceInfos 查询，代码未实现 |
| **偏差** | 代码实现与设计文档描述不一致 | 设计文档规定返回 Optional，代码返回 null |
| **超出** | 代码中存在设计文档未描述的行为 | 代码硬编码了设计文档未提及的超时值 |
| **质量缺陷** | 代码本身的问题 | 空指针风险、过长方法、重复逻辑 |

使用 AskUserQuestion 让用户确认发现类型分类。

## 2. 反向搜索设计文档

### 2.1 搜索策略（按优先级顺序执行）

**第一层：类名/包名精确匹配**
```
Grep <ClassName> in docs/ 目录
Grep <package.subpackage> in docs/ 目录
```

**第二层：方法名/字段名匹配**
```
Grep <methodName> in docs/ 目录
Grep <fieldName> in docs/ 目录
```

**第三层：业务概念关键词**
从代码注释、变量名中提取业务关键词：
```
Grep <domain-keyword> in docs/ 目录
```

**第四层：模块文档映射**
若项目有 `plans/` 目录或开发计划文档，根据包路径映射到对应模块设计文档。

### 2.2 确认关联文档

将搜索结果按匹配度排序，呈现给用户确认：

```
📋 根据代码文件 <path>，定位到以下设计文档：

| # | 文档 | 章节 | 匹配方式 |
|---|------|------|---------|
| 1 | docs/.../03a-core-models.md | §2.3 | 精确匹配类名 |
| 2 | docs/.../05-external-dependency.md | §3.1 | 匹配方法名 |
| 3 | docs/.../02-architecture-overview.md | §1.2 | 模块映射 |

是否正确？是否需要补充其他文档？
```

### 2.3 边缘场景

- **无匹配设计文档** → 询问用户：
  - A) 先创建设计文档再继续（推荐）
  - B) 跳过设计文档修补，仅生成代码修改计划（标记 🟡 警告：跳过同步）
- **命中过多（>5 个文档）** → 按匹配度只呈现 top 3-5，其余列为"次要关联"
- **模糊匹配不确定** → 使用 AskUserQuestion 让用户确认关联性

## 3. 代码-设计文档对比分析

### 3.1 构建对比表

| 对比维度 | 设计文档描述 | 代码实现 | 一致性 | 备注 |
|----------|------------|---------|--------|------|
| 类结构 | ... | ... | ✅/❌/⚠️ | |
| 方法签名 | ... | ... | ✅/❌/⚠️ | |
| 业务规则 | ... | ... | ✅/❌/⚠️ | |
| 异常处理 | ... | ... | ✅/❌/⚠️ | |
| 参数定义 | ... | ... | ✅/❌/⚠️ | |

### 3.2 差异定级

- 🔴 **阻塞**：核心行为缺失或矛盾
- 🟡 **警告**：次要行为偏差或边界处理不一致
- 🟢 **建议**：命名/风格/注释层面的差异

## 4. /grill-me 上下文构建

代码发现模式中，/grill-me 的输入上下文应同时包含：

### 4.1 必呈内容

1. **代码现状**：关键代码片段（方法体、类结构、字段定义）
2. **设计文档描述**：关联章节原文
3. **差异分析**：对比表中的不一致项及定级
4. **影响分析**：
   - 若以设计文档为准 → 需修改哪些代码
   - 若以代码为准 → 需修改哪些设计文档章节

### 4.2 追问重点

- 以设计文档还是代码为准？（推荐：设计文档为准）
- 如果以设计文档为准，代码修改的优先级和范围？
- 设计文档本身是否需要补充/更新？（设计文档可能遗漏了新需求）
- 变更是否影响其他模块？需要同步修改哪些设计文档章节？

## 6. 交接 /triage-issue 机制

### 6.1 轻量变更判断

代码修改计划生成后，先判断变更是否属于轻量：

| 判断条件 | 量化标准 |
|----------|---------|
| 文件数量 | ≤ 1 个源码文件 |
| 行为变更 | 无方法签名变更、无新增/删除公共方法 |
| 影响范围 | 仅当前类内部，无跨模块调用链 |

**全部满足** → 轻量，直接建议 /tdd，跳过交接。

### 6.2 交接触发

非轻量变更时，使用 AskUserQuestion 询问：

```
代码修改计划已生成。下一步操作：
  A) 交接 /triage-issue 创建 issue（推荐）
  B) 直接创建 issue（决策已确认，跳过重复探索）
  C) 仅输出计划，稍后手动处理
  D) 计划需要调整
```

### 6.3 交接执行

用户选择 A 时，使用 Skill 工具调用 /triage-issue：

```
/triage-issue

问题描述：
<代码修改计划的「问题」章节>

补充资料（来自 /resolve-finding 分析，决策已确认，可跳过根因分析步骤）：
- 根因分析：<代码修改计划的「根因分析」章节>
- TDD 修复计划：<代码修改计划的「TDD 修复计划」章节>
- 验收标准：<代码修改计划的「验收标准」章节>
- 决策记录：<grill-decisions 文件路径 或 内联决策表>
- 已修补的设计文档：<文档路径列表>
- 受影响模块：<模块和文件列表>
```

用户选择 B 时，直接使用 `gh issue create`（GitHub）或 `glab issue create`（GitLab）创建 issue，将代码修改计划内容填充为 issue body。平台判断规则见 CLAUDE.md。

### 6.4 交接后行为

- resolve-finding 不再参与后续流程
- /triage-issue 按自身流程执行探索诊断（验证代码修改计划）→ 确认修复方案 → 创建 issue
- /triage-issue 应参考决策记录获取上下文
- 注意：/triage-issue 会执行自己的代码库探索，交接内容作为起点而非最终结论

### 6.5 交接前检查

- [ ] 设计文档已修补完毕（设计文档优先于代码）
- [ ] 代码修改计划完整（包含 TDD 循环和验收标准）
- [ ] /grill-me 决策记录已保存（如用户选择保存）
- [ ] 连锁检查已完成（无未处理的连锁影响）
