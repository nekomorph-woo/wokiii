# 通用步骤

> 反向修补、连锁检查、状态更新的通用实现，所有模式共用。

## 反向修补

按 /grill-me 决策修改设计文档。

### 修补方式

| 输出目标 | 行为 |
|----------|------|
| 设计文档 | 展示 diff → 用户确认 → 写入文件 |
| 控制台 | 仅输出修改建议，不写文件 |

通用发现模式中由用户在输入阶段选择；审阅报告模式和代码发现模式默认输出到设计文档。

### 展示修改 diff

```
📄 <设计文档路径>

--- 修改前
+++ 修改后
- 旧描述
+ 新描述（基于决策结论）
```

涉及多个文件时逐个展示。

### 用户确认

使用 AskUserQuestion 展示 diff，用户确认后写入文件。

### 多文件修补顺序

若决策涉及多个设计文档的修改，按依赖顺序依次修补：
1. 先修核心定义（模型、接口）
2. 再修引用这些定义的模块文档
3. 最后修总览和关系文档

---

## 连锁检查

### 检查方法

- Grep 搜索其他文档中引用被修改章节的位置
- 确认这些引用是否需要同步更新

### 连锁深度控制

连锁检查最多 2 级。第 2 级之后的连锁项列出但不深入处理，由用户决定后续行动。
