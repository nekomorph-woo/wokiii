# 检查项详细规范

## 1. 接口一致性

**检查内容**：

模块 A 调用模块 B 的接口时，两者的定义是否一致。

**检查维度**：

| 维度 | 检查方式 |
|------|----------|
| 参数名和类型 | A 传的参数是否匹配 B 的定义 |
| 返回值类型 | A 期望的返回是否匹配 B 的声明 |
| 异常处理 | A 是否处理了 B 可能抛出的异常 |
| 命名一致性 | 同一概念在不同模块中是否使用相同术语 |

**严重度**：

- 🔴 参数类型不匹配、返回值类型不匹配
- 🟡 异常未处理、命名不一致
- 🟢 文档格式不一致（如描述风格差异）

## 2. 依赖方向

**检查内容**：

模块间依赖是否单向、是否违反分层约束。

**检查维度**：

| 维度 | 检查方式 |
|------|----------|
| 循环依赖 | 构建依赖图，检测环 |
| 分层违反 | 核心模块是否依赖了外围模块 |
| 隐式依赖 | design.md 中引用但 _registry.md 未声明的依赖 |

**严重度**：

- 🔴 循环依赖
- 🟡 分层违反、隐式依赖

## 3. 跨模块遗漏

**检查内容**：

设计锚点是否被全部响应，用户故事是否有模块负责实现。

**检查维度**：

| 维度 | 检查方式 |
|------|----------|
| 锚点覆盖 | 逐条检查设计锚点，哪些模块响应了哪些锚点 |
| 故事覆盖 | 逐条检查用户故事，哪些模块负责实现 |
| 孤立模块 | 是否有模块不被任何用户故事需要 |
| 缺失模块 | 是否有用户故事没有对应模块负责 |

**严重度**：

- 🔴 设计锚点无模块响应、用户故事无模块负责
- 🟡 存在孤立模块、模块职责重叠
- 🟢 验收标准与接口定义不完全对齐
