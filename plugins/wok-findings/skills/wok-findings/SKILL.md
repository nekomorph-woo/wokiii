# 代码探索

探索现有代码的设计约束和质量现状，通过对话输出发现，大探索时自动持久化为 `_findings.md`。

## 快速开始

1. 询问用户要探索的代码区域
2. 多维度探索代码库
3. 输出发现和设计约束

## 工作流程

### 1. 确定范围

询问用户：

- 要探索哪些模块/目录/文件？
- 关注什么方面？（架构、设计模式、代码质量、依赖关系）
- 是否有特定的疑问或假设？

### 2. 代码探索

使用 Agent 子代理并行探索以下维度（详见 reference/exploration-dimensions.md）：

| 维度 | 目标 | 工具 |
|------|------|------|
| 架构 | 模块边界、分层、依赖方向 | Glob + Grep + Read |
| 模式 | 设计模式、约定、惯例 | Grep + Read |
| 约束 | 技术栈限制、外部依赖、硬性要求 | Read 配置文件 + Grep |
| 痛点 | 代码坏味道、耦合点、脆弱区域 | Grep + Read |
| 术语 | 领域概念、命名约定 | Grep |

### 3. 输出发现

直接在对话中输出，不生成文件：

```
## 探索发现

**范围**：<探索的代码区域>
**架构概览**：<模块关系简述>

### 设计约束
- 约束 1

### 现有模式
- 模式 1

### 代码→文档映射

| 代码区域 | 对应设计文档 | 状态 |
|----------|-------------|------|
| src/auth/ | plans/auth/modules/auth-core/design.md | ✅ 已有 |
| src/payment/ | 无 | ⚠️ 无设计文档 |

### 潜在问题
- 🟡 问题 1
- 🔴 问题 2

### 对后续设计的影响
- <建议>
```

代码→文档映射说明：

- 使用 4 层反向搜索定位：类名精确匹配 → 方法名匹配 → 业务概念 → 模块映射
- 标注每个代码区域是否有对应的设计文档
- 无设计文档的代码区域标记为 ⚠️ — 后续 `wok-design` 应优先覆盖

### 4. 持久化发现（可选）

评估探索规模：

| 条件（满足任一） | 判定 |
|------|------|
| 探索了 3+ 个维度 | 大探索 |
| 发现 5+ 个设计约束 | 大探索 |
| 代码→文档映射存在 ⚠️ 项 | 大探索 |
| 均不满足 | 小探索 |

**大探索**：自动生成 `_findings.md`。

保存路径判断（按优先级）：
1. 对话上下文中用户提及的 system-name → `plans/<system-name>/_findings.md`
2. `plans/` 下仅一个子目录 → 使用该目录
3. 均不满足 → AskUserQuestion 询问保存位置

**小探索**：使用 AskUserQuestion 询问用户是否需要生成。

**_findings.md 输出格式**：

```markdown
---
status: draft
intent: reference
scope: global
depends: []
changed: 初始版本
---

# 代码探索发现

> **做什么**：记录代码库现状基线，为后续设计提供输入
> **怎么做**：多维度并行探索
> **阻塞**：无

## 探索范围

<探索的代码区域>

## 架构概览

<模块关系简述>

## 设计约束

<约束列表>

## 现有模式

<模式列表>

## 代码→文档映射

| 代码区域 | 对应设计文档 | 状态 |
|----------|-------------|------|

## 潜在问题

<问题列表>

## 对后续设计的影响

<建议>
```

生成后告知文件路径。下游 `wok-idea` 会自动读取此文件作为上下文。

## 定位

wok-findings 是管道的可选入口。当功能涉及已有代码时，在 `wok-define` 之前运行，建立现状基线。

也可独立使用：在任何需要理解代码库的时候。

## 约束

- **DO NOT** 修改代码 — 纯探索 + 可选产出物
- **DO NOT** 提供修复方案 — 发现问题即可，修复属于后续步骤
