# 基于本体论的软件迭代策略方法论

## 0. 摘要

本方法论用于指导复杂软件系统的需求迭代、领域建模、版本拆分与设计演进。

它的核心观点是：

> 软件系统不应被视为功能列表的堆叠，而应被视为一个可描述、可运行、可约束、可演化的领域世界。

传统软件迭代通常以功能为中心：新增页面、新增按钮、新增接口、新增流程。短期看，这种方式推进速度很快；长期看，它容易导致对象边界模糊、状态混乱、规则散落、流程重复、上下文不可追踪，最终形成一个“功能很多但系统不理解自己”的应用。

本体论驱动的软件迭代方法则将重点从“做什么功能”转移到“这个系统世界里有什么对象、对象之间有什么关系、对象能发生什么行为、行为受什么规则约束、行为之后世界状态如何改变”。

因此，软件迭代的基本单位不再是孤立功能，而是：

> 对象、关系、行为、规则共同构成的能力闭环。

---

## 1. 方法论的基本立场

### 1.1 软件不是功能集合，而是领域世界

一个软件系统表面上由页面、按钮、接口、数据表组成，但真正决定系统复杂度的不是这些表层结构，而是软件内部如何表达它所服务的世界。

例如，一个系统可能包含“创建、审核、发布、撤回、归档”等功能。如果只按功能实现，每个功能都可能拥有自己的字段、状态、判断逻辑和异常处理。随着版本增加，系统会逐渐出现逻辑分裂：

```text
创建逻辑在 A 模块
审核逻辑在 B 模块
状态判断在 C 接口
权限规则在 D 页面
异常处理在 E 服务
```

这时软件仍然能运行，但它已经不再拥有统一的世界模型。

本体论方法要求先回答：

```text
这个系统世界中有哪些核心对象？
这些对象如何关联？
这些对象有什么生命周期？
哪些行为可以改变对象状态？
哪些规则约束这些行为？
```

只有这些问题被稳定回答后，功能实现才不会变成到处贴补丁的行为艺术。人类工程史的一大半，就是把补丁贴到补丁上，然后给它起名叫“架构演进”。

### 1.2 功能是表象，对象才是稳定核心

功能经常变化，但对象相对稳定。

用户可能今天需要“审批”，明天需要“批量审批”，后天需要“自动审批”，再后来需要“条件审批”。如果系统只围绕审批功能设计，那么每次需求变化都会产生新的流程分支。

但如果系统先识别出稳定对象：

```text
申请
审批记录
审批人
审批规则
审批结果
状态变更
```

那么后续功能变化只是这些对象之间关系和行为的扩展。

因此，本方法论强调：

> 不要先问“要做什么功能”，而要先问“这个功能背后暴露了什么对象和关系”。

### 1.3 关系比字段更重要

传统设计容易把对象设计成字段容器：

```text
对象 = 一堆属性
```

本体论设计则认为：

```text
对象 = 属性 + 关系 + 行为 + 约束 + 生命周期
```

字段只能描述对象自己，而关系描述对象处在什么世界中。

一个对象孤立存在时，它只是数据；当它与其他对象产生关系时，它才进入系统世界。

例如：

```text
订单
订单项
用户
支付记录
发票
物流单
售后单
```

如果只看字段，它们是几张表。如果看关系，它们共同构成一个交易世界：

```text
用户创建订单
订单包含订单项
订单触发支付
支付完成后生成履约任务
履约完成后可开发票
订单完成后可进入售后
```

这套关系才是系统真正的骨架。

### 1.4 行为和约束决定系统是否“能运行”

对象和关系只是静态结构，行为和约束才让世界运行起来。

一个本体论软件系统必须回答：

```text
对象能做什么？
什么状态下能做？
谁可以做？
做完以后产生什么结果？
失败以后如何处理？
```

例如，一个“文档”对象不仅有标题、正文、作者、时间，还应该有行为：

```text
创建
编辑
提交审核
批准
驳回
发布
归档
```

同时有约束：

```text
只有草稿状态可以编辑
只有待审核状态可以批准
已归档对象不可再次发布
发布前必须通过审核
```

这些行为和约束构成了系统的运行规则。

没有规则的系统，只是自由落体的 JSON。看起来很灵活，落地时很响。

## 2. 核心概念体系

本方法论建议将软件设计拆分为六类核心概念。

### 2.1 Entity：实体对象

实体对象是系统世界中可以被识别、追踪和操作的核心存在。

实体对象通常具备：

```text
唯一标识
类型
状态
生命周期
可被引用
可参与关系
可触发行为
```

示例：

```text
用户
订单
合同
任务
项目
设备
审批单
报告
事件
规则
策略
记录
```

判断某个概念是否应该成为实体对象，可以使用以下标准：

| 判断问题 | 如果答案为是，倾向于建模为实体 |
|---|---|
| 它是否需要独立存在？ | 是 |
| 它是否有自己的状态？ | 是 |
| 它是否会被多个功能引用？ | 是 |
| 它是否有生命周期？ | 是 |
| 它是否会与其他对象建立关系？ | 是 |
| 它是否需要被审计、追踪、查询？ | 是 |

如果一个概念只是临时输入，通常不应成为实体对象。

### 2.2 Attribute：属性

属性用于描述实体自身的静态或半静态信息。

例如：

```text
名称
编号
创建时间
更新时间
优先级
类型
描述
金额
标签
来源
```

但属性设计必须克制。

本方法论建议将属性分为三层：

#### 第一层：身份属性

用于识别对象。

```text
id
type
name/title
createdAt
updatedAt
```

#### 第二层：运行属性

用于驱动系统行为。

```text
status
priority
owner
stage
version
```

#### 第三层：扩展属性

用于补充描述，不直接驱动核心流程。

```text
tags
notes
metadata
extra
customFields
```

属性晋升原则：

> 只有当一个属性被查询、过滤、排序、权限判断、状态机、规则引擎或多个能力闭环稳定依赖时，才应成为正式字段。

否则，它应该暂时留在扩展区。

这条很重要。不然每一次“以后可能有用”都会变成数据库里一根永久的钉子，直到系统像刺猬一样无法拥抱任何变化。

### 2.3 Relation：关系

关系描述实体对象之间的连接方式。

常见关系包括：

```text
属于
包含
依赖
产生
引用
阻塞
触发
派生
替代
继承
影响
关联
授权
负责
```

关系设计要回答：

```text
谁与谁有关？
是什么关系？
关系是否有方向？
关系是否有强度？
关系是否有生命周期？
关系是否会影响行为？
```

示例：

```text
用户 创建 订单
订单 包含 商品项
合同 依赖 审批单
审批单 产生 审批记录
异常 阻塞 流程
规则 约束 操作
```

关系应被视为一等公民，而不是对象字段中的附属 ID。

因为真正复杂的软件系统，复杂性通常不来自单个对象，而来自对象之间不断增加的依赖网络。

### 2.4 Action：行为

行为是对象可以发生的操作。

行为不是普通函数名，而是领域世界中的动作。

例如：

```text
创建
提交
审批
拒绝
撤回
发布
暂停
恢复
关闭
归档
复制
转移
分配
确认
解决
验证
```

每个行为都必须定义：

```text
作用对象
输入参数
前置条件
执行过程
结果产物
状态变化
副作用
失败处理
审计记录
```

行为设计模板：

```text
Action: submit

Target Entity:
- Document

Preconditions:
- Document.status = draft
- User has submit permission
- Required fields are complete

Effects:
- Document.status -> pending_review
- Create ReviewRequest
- Append AuditLog

Failure:
- Missing required field
- Permission denied
- Invalid state
```

行为是对象从静态数据变成运行世界的关键。

### 2.5 Constraint：约束

约束定义什么事情可以发生，什么事情不能发生。

约束类型包括：

```text
状态约束
权限约束
数据完整性约束
业务规则约束
时间约束
数量约束
依赖约束
安全约束
合规约束
```

示例：

```text
已发布内容不可直接编辑
未付款订单不可发货
存在阻塞问题时不可关闭任务
审批人不能审批自己提交的申请
超过截止时间后只能走补充流程
```

约束是本体论系统中最容易被忽视、但最能体现世界模型价值的部分。

普通系统经常把约束散落在页面、接口和数据库校验里。本体论方法要求将约束显式表达出来，使系统可以解释：

```text
为什么不能执行这个操作？
当前缺少什么条件？
解除约束需要什么行为？
```

### 2.6 State：状态与生命周期

状态描述对象当前处于生命周期的哪个阶段。

一个对象如果没有状态，就很难形成清晰的行为规则。

状态设计要回答：

```text
对象从哪里开始？
中间有哪些阶段？
哪些状态是终态？
哪些行为触发状态迁移？
哪些迁移是不允许的？
异常状态如何恢复？
```

示例：

```text
draft
pending
active
paused
blocked
completed
failed
archived
```

状态机设计示例：

```text
draft -> submit -> pending
pending -> approve -> active
pending -> reject -> draft
active -> pause -> paused
paused -> resume -> active
active -> complete -> completed
completed -> archive -> archived
```

状态机的价值在于，它让系统能够判断：

```text
当前对象允许发生什么？
不允许发生什么？
发生之后会去哪里？
```

这比“到处 if else 判断状态”要可靠得多。虽然人类显然很喜欢把 if else 写成现代洞穴壁画。

## 3. 本体论驱动的需求分析方法

### 3.1 从需求文本中提取领域元素

面对一个需求，不要急着拆页面、接口和数据库表。

先提取四类元素：

```text
名词：候选对象
动词：候选行为
形容词/状态词：候选状态或属性
条件句：候选约束
```

示例需求：

```text
用户可以提交申请，管理员审核后申请生效。如果申请信息不完整，则不能提交。用户可以撤回未审核的申请。
```

提取结果：

| 类型 | 内容 |
|---|---|
| 候选对象 | 用户、申请、管理员、审核 |
| 候选行为 | 提交、审核、生效、撤回 |
| 候选状态 | 未审核、已生效、信息不完整 |
| 候选约束 | 信息不完整不能提交；未审核才能撤回 |

然后再进一步判断哪些概念需要成为实体，哪些只是属性或状态。

### 3.2 用五问法审查每个对象

每个候选对象必须回答五个问题：

```text
1. 它是什么？
2. 它和谁有关？
3. 它能做什么？
4. 它什么时候能做？
5. 它做完以后会改变什么？
```

对应本体论结构：

| 问题 | 对应概念 |
|---|---|
| 它是什么？ | Entity |
| 它和谁有关？ | Relation |
| 它能做什么？ | Action |
| 它什么时候能做？ | Constraint |
| 它做完以后会改变什么？ | Effect / State Transition |

如果一个对象无法回答这些问题，它可能只是字段、视图模型或临时概念，不应过早成为核心实体。

### 3.3 区分领域对象与界面对象

一个常见错误是把页面上的东西误认为领域对象。

例如：

```text
列表
详情页
弹窗
表单
筛选器
标签页
卡片
按钮
```

这些通常是界面对象，不是领域对象。

领域对象应该是业务世界中真实存在或被系统稳定追踪的东西。

对比：

| 界面对象 | 领域对象 |
|---|---|
| 订单详情页 | 订单 |
| 审批弹窗 | 审批行为 / 审批记录 |
| 筛选器 | 查询条件 |
| 卡片 | 任务 / 项目 / 记录 |
| 发布按钮 | 发布行为 |

界面可以变化，领域对象应该保持稳定。

### 3.4 区分动作与对象

另一个常见错误是把动作名误建模为对象。

例如：

```text
审核
发布
同步
导入
导出
```

这些通常是行为，不一定是对象。

但有些动作会产生可追踪结果，此时可以派生出对象。

例如：

```text
审核 -> 审核记录
导入 -> 导入任务 / 导入批次
同步 -> 同步任务 / 同步日志
发布 -> 发布记录 / 发布版本
```

判断规则：

> 如果动作执行后需要被查询、追踪、审计、回滚或参与后续流程，则它的结果应该建模为对象。

## 4. 本体论驱动的版本迭代策略

### 4.1 版本不按功能堆叠，而按世界成熟度推进

传统版本规划通常这样：

```text
v1.0 做基础功能
v1.1 做高级功能
v1.2 做管理功能
v1.3 做统计功能
```

本体论驱动版本规划应该这样：

```text
v1.0 建立核心对象
v1.1 建立对象关系
v1.2 建立对象行为
v1.3 建立状态机和约束
v1.4 建立能力闭环
v1.5 建立解释能力
v1.6 建立自动化能力
```

也就是说，版本迭代的目标不是“功能越来越多”，而是“系统对领域世界的理解越来越深”。

### 4.2 能力闭环是版本拆分的基本单位

一个版本应该交付一个完整能力闭环，而不是一组零散功能。

能力闭环定义：

> 一个用户目标从开始到结束，经过对象创建、关系建立、行为执行、状态变化、规则校验、结果产出，能够形成可验证结果的最小流程。

例如：

```text
创建申请
提交申请
审核申请
申请生效
生成审核记录
```

这是一个闭环。

而下面这些不是闭环：

```text
做申请列表
做审核按钮
做状态字段
做管理员页面
```

这些只是零件。

零件当然需要，但不能以零件作为版本目标。否则最后你会得到一个零件仓库，而不是一个能跑的车。

### 4.3 MVP 应该验证核心世界，而不是覆盖完整功能

MVP 的目标不是“功能少”，而是“用最小实现验证领域世界是否成立”。

一个好的 MVP 应该包含：

```text
最少对象
最少关系
最少行为
最少状态
最少约束
一个完整闭环
```

MVP 不应该包含：

```text
复杂配置
多角色扩展
高级筛选
批量操作
统计报表
主题皮肤
个性化设置
复杂权限矩阵
```

除非它们是核心闭环不可缺少的部分。

MVP 判断标准：

| 问题 | 判断 |
|---|---|
| 没有这个对象，核心闭环是否无法表达？ | 是则保留 |
| 没有这个关系，流程是否无法推进？ | 是则保留 |
| 没有这个行为，用户目标是否无法完成？ | 是则保留 |
| 没有这个约束，系统是否会产生错误状态？ | 是则保留 |
| 没有这个页面美化，闭环是否仍可验证？ | 是则砍掉 |

### 4.4 版本演进模型

建议将系统迭代划分为七个成熟阶段。

#### 阶段 1：对象识别

目标：

```text
确定系统中最核心的实体对象。
```

产出：

```text
核心对象清单
对象定义
对象基本属性
对象边界说明
```

重点问题：

```text
这个系统到底管理什么？
哪些对象必须长期存在？
哪些对象有独立生命周期？
哪些对象只是临时输入？
```

#### 阶段 2：关系建模

目标：

```text
确定核心对象之间的稳定关系。
```

产出：

```text
关系图
关系类型定义
关系方向
关系约束
```

重点问题：

```text
对象之间如何连接？
谁依赖谁？
谁产生谁？
谁影响谁？
谁拥有谁？
```

#### 阶段 3：行为定义

目标：

```text
定义对象可以发生的领域行为。
```

产出：

```text
行为清单
行为输入
行为前置条件
行为结果
行为失败情况
```

重点问题：

```text
对象能做什么？
用户能对对象做什么？
系统能自动对对象做什么？
行为会产生什么新对象或新关系？
```

#### 阶段 4：状态机与生命周期

目标：

```text
明确对象在不同阶段的状态变化。
```

产出：

```text
状态定义
状态迁移图
非法迁移规则
异常恢复规则
```

重点问题：

```text
对象有哪些状态？
哪些状态是中间态？
哪些状态是终态？
什么行为触发状态变化？
哪些状态下禁止某些行为？
```

#### 阶段 5：约束与规则

目标：

```text
让系统能够判断什么可以发生，什么不可以发生。
```

产出：

```text
业务规则
权限规则
数据规则
依赖规则
异常规则
```

重点问题：

```text
行为执行前必须满足什么条件？
哪些对象状态会阻止流程继续？
哪些关系会产生限制？
规则失败时如何解释？
```

#### 阶段 6：能力闭环

目标：

```text
将对象、关系、行为、状态、约束组合成完整用户目标。
```

产出：

```text
端到端流程
验收标准
闭环测试用例
异常路径
```

重点问题：

```text
用户目标是否能完整完成？
系统是否能追踪整个过程？
结果是否可验证？
异常是否可恢复？
```

#### 阶段 7：解释与自动化

目标：

```text
让系统能够解释当前局势，并支持进一步自动化。
```

产出：

```text
阻塞原因解释
下一步动作推荐
上下文装配规则
自动化执行策略
审计记录
```

重点问题：

```text
当前发生了什么？
为什么不能继续？
下一步允许做什么？
系统可以自动做什么？
哪些地方必须人工介入？
```

## 5. 对象设计原则

### 5.1 对象必须拥有清晰边界

一个对象边界不清，会导致职责膨胀。

错误示例：

```text
User 对象包含：
- 用户资料
- 登录凭证
- 权限配置
- 订阅信息
- 账单信息
- 行为统计
- 通知设置
```

这会导致 User 成为上帝对象。

更好的拆分：

```text
User
Credential
Role
Subscription
BillingAccount
UserPreference
UserActivity
```

判断对象是否过大，可以看：

```text
它是否拥有多个不同生命周期？
它是否被不同模块以不同方式修改？
它是否承载了互不相关的规则？
它是否频繁因为不同原因变化？
```

如果答案为是，就应该拆分。

### 5.2 对象不应因为页面存在而存在

不要因为有一个页面，就创造一个领域对象。

错误：

```text
DashboardEntity
ListEntity
DetailEntity
FormEntity
```

正确：

```text
页面展示多个领域对象的视图。
```

界面层可以有 ViewModel，但不要污染领域模型。

### 5.3 对象应围绕生命周期设计

一个对象是否成立，关键看它是否有生命周期。

例如：

```text
草稿 -> 待审核 -> 已通过 -> 已发布 -> 已归档
```

生命周期越清晰，对象越有建模价值。

如果一个概念没有生命周期，可能只是属性或枚举。

### 5.4 对象应避免过早泛化

不要一开始就设计万能对象：

```text
Entity
Node
Item
Resource
Object
Thing
```

这些名字看起来很高级，实际经常只是“我还没想清楚”的遮羞布。软件工程最古老的化妆品。

早期应该使用具体领域对象。

当多个具体对象出现稳定共性后，再抽象基类或通用模型。

抽象顺序：

```text
具体对象
重复结构
稳定共性
抽象模型
```

不要反过来。

---

## 6. 属性设计原则

### 6.1 先少后多

早期对象属性应尽量少。

推荐初始属性：

```text
id
type
title/name
status
createdAt
updatedAt
metadata
```

然后根据闭环需要逐步晋升字段。

### 6.2 属性晋升规则

一个字段从 metadata 晋升为正式字段，应满足至少一个条件：

```text
需要被频繁查询
需要被排序或筛选
参与状态机
参与权限判断
参与业务规则
参与跨对象关系
被多个能力闭环依赖
需要审计或索引
```

不满足这些条件的字段，不应过早正式化。

### 6.3 属性必须有语义，不只是数据形状

字段设计不应只关注类型，还要关注语义。

不要只写：

```text
status: string
```

而要定义：

```text
status 表示对象生命周期阶段
允许值：
- draft
- pending
- active
- completed
- archived

每个状态含义：
draft：尚未提交
pending：等待处理
active：已生效
completed：已完成
archived：不可变历史状态
```

字段没有语义定义，就会被不同开发者按自己的想象使用。然后系统进入“每个人都没错，但系统完蛋了”的优雅状态。

---

## 7. 关系设计原则

### 7.1 关系要显式化

不要把重要关系藏在字段里。

弱设计：

```text
task.artifactId
```

强设计：

```text
Task --produces--> Artifact
```

显式关系可以支持：

```text
关系查询
影响分析
上下文装配
权限推导
依赖判断
审计追踪
自动化推理
```

### 7.2 关系必须有类型

不要只有“关联”。

错误：

```text
A relatedTo B
```

正确：

```text
A dependsOn B
A produces B
A blocks B
A owns B
A approves B
A references B
```

关系类型越明确，系统越能理解当前局势。

### 7.3 关系可以有属性

有些关系本身需要属性。

例如：

```text
User --assignedTo--> Task
```

这条关系可能有：

```text
assignedAt
assignedBy
role
expiresAt
```

不要把所有信息都塞进实体。关系本身也可以承载语义。

---

## 8. 行为设计原则

### 8.1 行为必须属于领域，而不是接口

不要这样定义行为：

```text
POST /api/updateStatus
```

要这样定义行为：

```text
approveApplication
rejectApplication
publishDocument
archiveProject
resolveIssue
```

接口是实现形式，行为是领域语义。

### 8.2 每个行为都必须有前置条件和结果

行为模板：

```text
行为名称：
目标对象：
参与对象：
输入参数：
前置条件：
执行结果：
状态变化：
产生对象：
产生关系：
失败情况：
审计记录：
```

没有前置条件的行为容易造成非法状态。

没有结果定义的行为容易造成副作用失控。

### 8.3 行为优先于页面设计

一个功能是否成立，不应先看页面，而应先看行为。

例如“发布功能”应先定义：

```text
什么对象可以发布？
什么状态下可以发布？
谁可以发布？
发布后状态如何变化？
是否生成发布记录？
是否可以撤回？
撤回后如何处理？
```

然后才设计按钮和页面。

按钮只是行为入口，不是行为本身。

---

## 9. 约束设计原则

### 9.1 约束必须集中表达

约束不应散落在：

```text
前端按钮禁用
后端接口判断
数据库字段限制
定时任务逻辑
脚本校验
```

这些地方可以执行约束，但约束定义本身应集中管理。

否则，同一条规则会出现五种版本。人类称之为“历史原因”，机器称之为“请毁灭我”。

### 9.2 约束应能解释

好的约束系统不仅返回失败，还要说明原因。

弱返回：

```text
操作失败
```

强返回：

```text
无法发布，因为当前对象仍有 2 个未解决的阻塞项。
```

更强返回：

```text
无法发布：
1. 必填字段 owner 缺失
2. 当前状态为 draft，必须先提交审核
3. 存在一个 blocking 级别异常
允许的下一步操作：
- 补全字段
- 提交审核
- 解决异常
```

解释能力是本体论模型落地的重要标志。

---

## 10. 上下文装配原则

### 10.1 上下文不应靠全文堆砌

复杂系统中，不能每次操作都加载所有数据。

本体论方法要求根据对象关系动态装配上下文。

例如，当处理某个对象时，上下文应包括：

```text
当前对象
直接相关对象
上游依赖
下游影响
当前状态
可执行行为
阻塞约束
最近事件
相关规则
验收标准
```

这比“把所有历史信息都塞进去”更稳定、更经济，也更可解释。

### 10.2 上下文应服务于决策

上下文不是资料合集，而是为了回答：

```text
当前发生了什么？
为什么是这样？
下一步能做什么？
做了会影响谁？
风险是什么？
```

本体论模型的价值，不在于存了多少数据，而在于它能否为下一步动作提供结构化判断。

## 11. 需求迭代操作流程

下面是一套可直接用于需求分析和版本设计的流程。

### Step 1：收集需求

输入：

```text
用户故事
业务描述
问题反馈
新功能设想
异常案例
流程说明
```

输出：

```text
原始需求文本
```

### Step 2：提取候选元素

从需求中提取：

```text
候选对象
候选属性
候选关系
候选行为
候选状态
候选约束
候选产物
```

### Step 3：判断对象边界

对每个候选对象进行判断：

```text
是否独立存在？
是否有生命周期？
是否被多处引用？
是否参与关系？
是否需要审计？
是否有行为？
```

输出：

```text
核心对象
辅助对象
临时概念
界面概念
```

### Step 4：建立关系图

定义对象之间的关系：

```text
A owns B
A creates B
A dependsOn B
A blocks B
A produces B
A approves B
A references B
```

输出：

```text
关系清单
关系图
关系约束
```

### Step 5：定义行为

为核心对象定义行为：

```text
行为名称
目标对象
前置条件
执行结果
状态变化
失败情况
```

输出：

```text
行为规格表
```

### Step 6：定义状态机

为有生命周期的对象定义：

```text
状态集合
初始状态
终态
允许迁移
非法迁移
异常恢复
```

输出：

```text
状态机定义
```

### Step 7：定义约束

定义所有关键约束：

```text
状态约束
权限约束
依赖约束
完整性约束
业务规则约束
```

输出：

```text
规则清单
失败解释
解除方式
```

### Step 8：组合能力闭环

将对象、关系、行为、状态、约束组合为一个可完成的闭环。

闭环格式：

```text
目标：
起点：
核心对象：
关键关系：
关键行为：
状态变化：
成功结果：
异常路径：
验收标准：
```

### Step 9：拆分版本

按闭环成熟度拆分版本：

```text
MVP：最小对象 + 最小关系 + 最小行为 + 最小闭环
v0.2：补充关键状态与约束
v0.3：增强异常路径
v0.4：增加解释能力
v0.5：增加自动化或高级扩展
```

### Step 10：验证世界一致性

每次迭代后检查：

```text
是否新增了重复对象？
是否出现关系缺失？
是否出现非法状态？
是否有规则散落？
是否有行为没有审计？
是否有对象职责膨胀？
是否有字段应该晋升？
是否有字段应该下沉？
```

## 12. 版本拆分决策表

| 需求类型 | 处理策略 |
|---|---|
| 新增核心对象 | 优先进入早期版本 |
| 新增对象关系 | 如果影响闭环，应优先实现 |
| 新增对象行为 | 如果完成用户目标必需，应进入当前版本 |
| 新增约束 | 如果避免错误状态必需，应立即实现 |
| 新增状态 | 如果生命周期无法表达，应实现 |
| 新增页面展示 | 如果不影响闭环，可延后 |
| 新增批量操作 | 通常延后 |
| 新增统计分析 | 通常在闭环稳定后实现 |
| 新增自动化 | 在规则稳定后实现 |
| 新增配置能力 | 在重复变化出现后实现 |

## 13. 反模式清单

### 13.1 Feature 堆叠

表现：

```text
版本规划全是功能清单。
```

风险：

```text
对象边界混乱
规则重复
状态不可控
```

修正：

```text
改为按能力闭环拆分版本。
```

### 13.2 字段膨胀

表现：

```text
对象字段越来越多，但行为和关系没有变清晰。
```

风险：

```text
对象职责过大
查询复杂
含义冲突
```

修正：

```text
字段分层，非核心字段进入 metadata，稳定后再晋升。
```

### 13.3 上帝对象

表现：

```text
一个对象承载多个生命周期和多个职责。
```

风险：

```text
难以维护
难以扩展
难以测试
```

修正：

```text
按生命周期和规则边界拆分对象。
```

### 13.4 隐式关系

表现：

```text
关系藏在某个 id 字段里，没有语义。
```

风险：

```text
无法做影响分析
无法解释上下文
无法推理依赖
```

修正：

```text
将关键关系显式建模。
```

### 13.5 规则散落

表现：

```text
前端、后端、数据库、脚本各有一套判断。
```

风险：

```text
规则不一致
异常难排查
行为不可解释
```

修正：

```text
集中定义约束，分层执行约束。
```

### 13.6 页面驱动建模

表现：

```text
先设计页面，再从页面反推对象。
```

风险：

```text
领域模型被 UI 污染
对象不稳定
复用困难
```

修正：

```text
先建领域对象，再设计页面表达。
```

## 14. 质量评估标准

一个本体论驱动的软件设计是否合格，可以用以下问题检查。

### 14.1 对象清晰度

```text
核心对象是否明确？
对象边界是否清楚？
对象是否有独立生命周期？
对象是否避免了承担过多职责？
```

### 14.2 关系完整度

```text
对象之间的关键关系是否显式？
关系是否有类型？
关系是否有方向？
关系是否能支持影响分析？
```

### 14.3 行为可控性

```text
行为是否有明确前置条件？
行为是否有明确结果？
行为是否会触发状态变化？
行为失败是否可解释？
```

### 14.4 状态一致性

```text
对象状态是否有限且明确？
状态迁移是否合法？
是否存在无法恢复的异常状态？
终态是否清晰？
```

### 14.5 约束集中度

```text
关键规则是否集中定义？
约束是否能解释失败原因？
约束是否被多个入口复用？
```

### 14.6 闭环完整性

```text
版本是否交付了完整用户目标？
是否只是交付零散功能？
是否有明确起点和终点？
是否能验证成功或失败？
```

### 14.7 演化能力

```text
新增需求是否能自然挂接到已有对象？
是否需要大量临时字段？
是否需要复制旧逻辑？
是否能通过关系和行为扩展？
```

## 16. 最终原则

可以将以下原则作为本体论软件迭代的最高指导规则。

```text
1. 不以功能清单作为设计起点，而以领域对象作为设计起点。

2. 不以页面作为系统边界，而以对象生命周期作为系统边界。

3. 不优先增加字段，而优先明确关系。

4. 不优先实现按钮，而优先定义行为。

5. 不将规则散落在代码各处，而将约束显式表达。

6. 不按零散功能拆版本，而按能力闭环拆版本。

7. 不追求一次性完整建模，而追求可验证、可演化的最小世界。

8. 不让对象无限膨胀，而根据生命周期和职责拆分对象。

9. 不让上下文依赖人工拼接，而通过对象关系动态装配。

10. 不只回答“系统有什么功能”，还要回答：
    - 世界里有什么？
    - 它们如何关联？
    - 它们能发生什么？
    - 什么情况下允许发生？
    - 发生后世界如何改变？
    - 当前为什么不能继续？
    - 下一步允许做什么？
```

---

## 17. 一句话定义

> 基于本体论的软件迭代，是一种以领域对象为核心、以对象关系为骨架、以行为和状态机为运行机制、以约束规则保证一致性、以能力闭环推进版本演进的软件设计方法。

它最终追求的不是“功能更多”，而是“系统越来越理解它所管理的世界”。

这才是本体论方法真正的价值。不是把对象属性填满，不是画一张漂亮的关系图然后供起来当神像，而是让软件在面对复杂变化时，仍然知道自己是谁、谁和谁有关、什么能发生、为什么不能发生，以及下一步应该发生什么。
