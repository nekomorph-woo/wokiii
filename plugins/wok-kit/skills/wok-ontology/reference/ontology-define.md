# 本体论 — 需求定义阶段指令

> 影响 wok-idea、wok-define 管道技能

## 需求提取规则

从需求文本中提取四类候选元素：

| 文本类型 | 提取目标 | 示例 |
|----------|----------|------|
| 名词 | 候选 Entity | 用户、申请、审批单 |
| 动词 | 候选 Action | 提交、审核、撤回 |
| 形容词/状态词 | 候选 State 或 Attribute | 未审核、已生效、信息不完整 |
| 条件句 | 候选 Constraint | "信息不完整则不能提交" |

## 五问审查

每个候选 Entity 必须回答五个问题，无法完整回答的降级为 Attribute 或临时概念：

| 问题 | 对应概念 | 降级条件 |
|------|----------|----------|
| 它是什么？ | Entity 定义 | 定义模糊 → 可能是 Attribute |
| 它和谁有关？ | Relation | 无关系 → 可能是独立配置 |
| 它能做什么？ | Action | 无行为 → 可能是枚举或常量 |
| 它什么时候能做？ | Constraint | 无前置条件 → 检查是否过于简单 |
| 它做完以后改变什么？ | Effect / State Transition | 无变化 → 可能是查询 |

## 对象判断规则

### 区分领域对象与界面对象

| 界面对象（不建模） | 领域对象（建模） |
|---------------------|------------------|
| 列表、详情页 | 订单、任务 |
| 弹窗、表单 | 审批行为、审批记录 |
| 筛选器、卡片 | 查询条件、项目 |
| 按钮、Tab | 发布行为 |

### 区分动作与对象

动作（审核、发布、同步、导入、导出）通常是 Action，不直接建模为 Entity。

**判断规则**：如果动作结果需要被查询、追踪、审计、回滚或参与后续流程，则派生为 Entity：
- 审核 → 审核记录
- 导入 → 导入批次
- 同步 → 同步日志
- 发布 → 发布版本

## 用户故事格式

用户故事必须标注涉及的核心 Entity：

```
作为 [角色]，对 [Entity] 执行 [Action]
前置：[Constraint]
预期：[State Change]
```

## 验收标准绑定

- 验收条件必须绑定到具体 Entity 的 Action 或 State 变更
- 每条可自动验证条件对应一个明确的对象行为断言
- ✅ "申请提交后 status 变为 pending_review"
- ❌ "提交功能正常工作"

## 版本规划

- 版本不按功能堆叠，按世界成熟度推进：对象→关系→行为→状态→约束→闭环
- MVP 验证核心世界：最少对象 + 最少关系 + 最少行为 + 一个完整闭环
- **MVP 排除**：复杂配置、多角色扩展、高级筛选、批量操作、统计报表、主题皮肤
