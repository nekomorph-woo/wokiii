# Rules 管理

管理项目的 `.claude/rules/` 配置文件，提供初始化、评估、改进、审计四项能力。

## 职责总览

| # | 职责 | 说明 |
|---|------|------|
| 1 | 初始化 | 拷贝 reference 模板到 `.claude/rules/` |
| 2 | 评估 | 对任意 rules 文件执行五维度评估，输出审查报告 |
| 3 | 改进 | 基于评估结果对 rules 进行重构 |
| 4 | 审计 | 检查多 rules 文件间的冲突/重叠/时效性 |

## 流程

### 0. 判断用户意图

用户触发时，根据上下文判断意图：

- 提到"初始化"/"新建"/"配置" → 执行初始化流程
- 提到"评估"/"审查"/"检查质量" → 执行评估流程
- 提到"改进"/"优化"/"重构 rules" → 执行改进流程
- 提到"审计"/"冲突检查"/"一致性" → 执行审计流程
- 意图不明确 → 询问用户选择哪项职责

## 2. 评估

对任意 `.claude/rules/` 下的 rules 文件执行五维度评估，输出结构化审查报告。

### 2.1 五维度评估模型

| 维度 | 核心标准 |
|------|----------|
| **Structure 结构** | 层次分明、模块化、路径隔离 |
| **Language 语言** | 祈使句、正向表述、无模糊词 |
| **Precision 精准度** | 具体可验证、含动机说明、含边界条件 |
| **Length 长度** | <200行、每文件单主题、token 经济 |
| **Maintenance 维护性** | 定期审计、无冲突规则、引用非复制 |

### 2.2 评估维度详解

#### Structure 结构

| 最佳实践 | 反模式 |
|----------|--------|
| Markdown 标题分组、bullet points、按主题拆分文件 | 单文件 400+ 行、大段文字墙 |
| 每文件单一职责 | 一个文件混合行为指令 + 参考资料 |

#### Language 语言

| 最佳实践 | 反模式 |
|----------|--------|
| 祈使句（"Use X"）、正向框架（"Do X" > "Don't do Y"） | 模糊词（"考虑使用"、"如果可能"）、被动语态 |
| 节名用功能描述（"交互规范"） | 风格化命名（"美学原则"） |

#### Precision 精准度

| 最佳实践 | 反模式 |
|----------|--------|
| 可验证指令、解释 WHY、1-3 个示例 | 抽象描述（"格式化代码" vs "用2空格缩进"） |
| 含边界条件和触发条件 | 无条件适用、无使用场景说明 |

#### Length 长度

| 最佳实践 | 反模式 |
|----------|--------|
| 单文件 < 200 行、rules 按主题拆分、token 经济 | 填塞所有规则到单文件、粘贴代码片段 |
| 删除 Claude 已知的参考资料（ASCII 字符表、Emoji 表） | 每次会话无条件加载低价值参考资料 |

#### Maintenance 维护性

| 最佳实践 | 反模式 |
|----------|--------|
| 引用 file:line 而非复制内容 | 粘贴模板代码、从不更新 |
| 定期审计冲突规则、"魔法短语测试" | 含过时规则、规则间语义重叠 |

### 2.3 执行步骤

1. 读取目标 rules 文件
2. 逐节按五维度扫描
3. 对每条规则判定 token 价值：🟢高（改变行为）/ 🟡中（有参考价值）/ 🔴低（噪音）
4. 输出结构化报告

### 2.4 评估报告格式

```
## [文件名] 评估报告

### 维度评分
| 维度 | 状态 | 说明 |
|------|------|------|
| 结构 | 🟢/🟡/🔴 | ... |
| 语言 | 🟢/🟡/🔴 | ... |
| 精准度 | 🟢/🟡/🔴 | ... |
| 长度 | 🟢/🟡/🔴 | ... |
| 维护性 | 🟢/🟡/🔴 | ... |

### 问题优先级矩阵
| 优先级 | 问题 | 改进建议 |
|--------|------|----------|
| 🔴 P0 立即修 | ... | ... |
| 🟡 P1 建议 | ... | ... |
| 🔵 P2 可选 | ... | ... |

### Token 价值统计
总行数: X | 高价值: Y (Z%) | 低价值: W (V%)
```

## 3. 改进

基于评估报告，对 rules 文件执行重构。

### 3.1 改进策略

| 策略 | 适用场景 | 示例 |
|------|----------|------|
| 删除噪音 | 🔴 低 token 价值内容 | ASCII 速查表、Emoji 表、工时估算 |
| 正向化 | 负向表述 | "**禁止**使用 X" → "使用 Y 代替 X" |
| 去模糊化 | 含模糊词 | "适当使用" → "仅在标注状态时使用" |
| 去重 | 语义重叠 | 多处表达相同约束 → 保留一处 |
| 去风格化 | 节名无信息量 | "美学原则" → "交互行为规范" |
| 精简化 | 形式化但无实际影响 | "置信度 X%" → 删除 |

### 3.2 执行步骤

1. 展示评估报告中的改进建议列表
2. 逐项确认用户是否同意该改进
3. 按确认结果执行重构
4. 输出变更摘要：`📄 文件 → 📝 变更内容 → 📊 影响范围`

---

## 4. 审计

检查 `.claude/rules/` 下所有 rules 文件间的一致性。

### 4.1 审计维度

| 维度 | 检查内容 |
|------|----------|
| 冲突 | 两条规则对同一行为给出矛盾指令 |
| 重叠 | 多条规则表达相同或高度相似的约束 |
| 时效性 | 引用的文件路径/工具/API 是否仍存在 |
| 作用域 | 是否可通过 `paths:` 实现按需加载以减少无条件加载 |
| 模型适配 | 对 Claude 4.6 是否存在过度强调（CRITICAL/MUST 过密） |

### 4.2 执行步骤

1. 扫描 `.claude/rules/` 下所有 `.md` 文件
2. 提取每条规则的核心行为指令
3. 交叉比对，标记冲突和重叠
4. 验证引用的文件路径是否仍存在
5. 输出审计报告

### 4.3 审计报告格式

```
## Rules 审计报告

### 文件清单
| 文件 | 行数 | 职责 |
|------|------|------|
| ... | ... | ... |

### 发现问题
| 类型 | 文件 A | 文件 B | 说明 |
|------|--------|--------|------|
| 🔴 冲突 | a.md:12 | b.md:5 | ... |
| 🟡 重叠 | a.md:3 | c.md:8 | ... |
| 🟡 过时 | d.md:7 | — | 引用路径不存在 |

### 统计
文件数: X | 总行数: Y | 冲突: Z | 重叠: W | 过时: V
```

## 关键认知

编写和评估 rules 时，遵循以下原则：

1. **guidance 不是 enforcement** — CLAUDE.md/rules 注入为 user message，不是 system prompt，不具备强制力
2. **指令遵循均匀退化** — 指令越多，所有指令的遵循度下降，优先保留高价值行为指令
3. **Lost in the Middle** — 长上下文中段信息遵循度最差，关键指令放开头/结尾
4. **正向表述 > 负向表述** — LLM 处理否定能力弱，存在 "ironic process effect"
5. **示例 > 规则** — 一个具体示例比十条抽象规则更有效
6. **模型适配** — Claude 4.6 无需 CRITICAL/MUST 过度强调，正常祈使句即可达成指令遵循

## Rules 编写原则

| 原则 | 说明 |
|------|------|
| **命令式** | 使用祈使句，直接指示动作 |
| **第三人称** | 不使用 "I will..." / "我会..." |
| **具体触发** | 明确 "Use when [条件]" |
| **精简示例** | 提供正确/错误对比 |
