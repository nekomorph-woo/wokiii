# 代码探索

探索现有代码的设计约束和质量现状，通过对话输出发现，大探索时自动持久化为 `_findings.md`。

## 快速开始

> 读取 `~/.claude/wok/resolve-system-name.md` 执行 system-name 解析。

1. 询问用户要探索的代码区域
2. 多维度探索代码库
3. 输出发现和设计约束

## 工作流程

### 1. 确定范围

询问用户：

- 要探索哪些模块/目录/文件？
- 关注什么方面？（架构、设计模式、代码质量、依赖关系）
- 是否有特定的疑问或假设？

**意图判断**：根据用户的问题和上下文判断是否需要深入追问：

| 用户意图 | 行为 |
|----------|------|
| 纯探索/了解代码（"看看这个模块怎么工作的"、"了解架构"） | 仅输出 findings，不调用 `/wok-grill-me` |
| 带方向性目的（建议、优化、讨论需求/方案/review/ideas） | 输出 findings 后，使用 `/wok-grill-me` 对关键发现深入追问 |

### 2. 代码探索

使用 Agent 子代理并行探索以下维度（详见 reference/exploration-dimensions.md）：

| 维度 | 目标 | 工具 |
|------|------|------|
| 架构 | 模块边界、分层、依赖方向 | Glob + Grep + Read |
| 模式 | 设计模式、约定、惯例 | Grep + Read |
| 约束 | 技术栈限制、外部依赖、硬性要求 | Read 配置文件 + Grep |
| 痛点 | 代码坏味道、耦合点、脆弱区域 | Grep + Read |
| 术语 | 领域概念、命名约定 | Grep |

### 3. 输出发现

直接在对话中输出，不生成文件：

```
## 探索发现

**范围**：<探索的代码区域>
**架构概览**：<模块关系简述>

### 设计约束
- 约束 1

### 现有模式
- 模式 1

### 代码→文档映射

| 代码区域 | 对应设计文档 | 状态 |
|----------|-------------|------|
| src/auth/ | .wok-plans/auth/modules/auth-core/design.md | ✅ 已有 |
| src/payment/ | 无 | ⚠️ 无设计文档 |

### 潜在问题

#### [OPEN] 🟡 问题 1
<问题描述>

#### [OPEN] 🔴 问题 2
<问题描述>

### 对后续设计的影响

- [ACTION] <建议 1>
```

代码→文档映射说明：

- 使用 4 层反向搜索定位：类名精确匹配 → 方法名匹配 → 业务概念 → 模块映射
- 标注每个代码区域是否有对应的设计文档
- 无设计文档的代码区域标记为 ⚠️ — 后续 `wok-design` 应优先覆盖

### 4. 持久化发现（延迟定型）

wok-findings 是唯一有歧义的入口（既可独立探索，也可作为 `feat-s-` 前置步骤）。

**不立即创建目录**。评估探索规模后，通过 AskUserQuestion 询问用户意图：

```
探索完成。接下来：
- 📂 保存为探索管道（exp-） → 创建 .wok-plans/exp-<name>/_findings.md
- 🛠️ 定义功能（feat-s-） → 转交 wok-define，_findings.md 存入 .wok-plans/feat-s-<name>/
- 🐛 修复问题（fix-） → 转交 wok-issue，产物存入 .wok-plans/fix-<name>/
```

**选"探索管道"**：
- 生成 `exp-` 前缀的 system-name（如 `exp-payment-module`）
- 创建 `.wok-plans/exp-<name>/` 目录并保存 `_findings.md`

**选"定义功能"**：
- 探索结果保留在对话上下文中
- 调用 `wok-define`，由 wok-define 创建 `.wok-plans/feat-s-<name>/` 目录
- `_findings.md` 和 `_define.md` 一并存入该目录

**选"修复问题"**：
- 探索结果保留在对话上下文中
- 调用 `wok-issue`，由 wok-issue 创建 `.wok-plans/fix-<name>/` 目录

评估探索规模（用于决定是否自动推荐持久化）：

| 条件（满足任一） | 判定 |
|------|------|
| 探索了 3+ 个维度 | 大探索 |
| 发现 5+ 个设计约束 | 大探索 |
| 代码→文档映射存在 ⚠️ 项 | 大探索 |
| 均不满足 | 小探索 |

**大探索**：直接弹出意图选择（不跳过持久化）。
**小探索**：额外提供"不保存"选项。

## 定位

wok-findings 是管道的可选入口。当功能涉及已有代码时，在 `wok-define` 之前运行，建立现状基线。

也可独立使用：在任何需要理解代码库的时候。

## 约束

- **DO NOT** 修改代码 — 纯探索 + 可选产出物
- **DO NOT** 提供修复方案 — 发现问题即可，修复属于后续步骤
- **DO NOT** 使用缩写标记（如 `D1`、`O1`），必须使用完整 `#### [OPEN]` / `- [ACTION]` 格式
