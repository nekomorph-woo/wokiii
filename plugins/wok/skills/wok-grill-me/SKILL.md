# Grill Me

对计划或设计进行系统性追问，逐层展开决策树并解决每个分支。

## 执行流程

1. **识别主题** - 确定要审视的计划或设计范围
2. **展开决策树** - 列出所有需要决策的分支点
3. **逐个决策** - 每次只处理一个决策点，直到得到用户同意才进入下一个
4. **确认共识** - 全部决策点完成后，回顾所有决策，确认理解一致
5. **询问保存** - 达成所有共识后，询问用户是否持久化决策记录

## 决策点处理

每个决策点必须得到用户的明确同意才能标记为已解决。处理流程：

### 场景 A：用户选择方案

使用 AskUserQuestion 发起决策：

```json
{
  "question": "<决策问题>",
  "header": "<决策点标题>",
  "options": [
    {"label": "方案 A（推荐）", "description": "理由简述"},
    {"label": "方案 B", "description": "理由简述"},
    {"label": "方案 C", "description": "理由简述"}
  ]
}
```

用户选择后 → 记录结果 → 进入下一个决策点。

### 场景 B：用户提问

用户选择"Other"并输入疑问而非方案时，**不跳过该决策点**：

1. 解答用户的疑问
2. 根据解答内容更新方案选项（若新信息改变了推荐）
3. 对同一决策点再次发起 AskUserQuestion
4. 重复直到用户做出选择

### 场景 C：代码库探索

决策可通过代码库/文档库解答时：

1. 自行探索，找到答案和来源位置
2. 向用户展示结论，标注来源：

```
**来源**: <file:line> 或 <文档路径>
**结论**: <从代码库/文档库得到的决策建议>
**理由**: <为什么这个方案适合>
```

3. 使用 AskUserQuestion 让用户确认或否决该结论
4. 用户同意 → 记录（标注来源）；用户否决 → 按场景 B 处理

## AskUserQuestion 规则

- 每次只对一个决策点发起 AskUserQuestion
- 选项之间必须有实质差异
- 推荐方案标注"（推荐）"
- 用户未做出决策选择时，**不进入下一个决策点**

## 追问原则

| 原则 | 说明 |
|------|------|
| **穷尽性** | 覆盖所有关键决策点，不遗漏分支 |
| **具体化** | 每次只处理一个决策点，完成后再进入下一个 |
| **不跳过** | 用户提问或疑惑时，解答后继续同一决策点，不跳到下一个 |
| **标注来源** | 代码库探索得到的结论必须标注来源位置 |
| **先探后问** | 能从代码库获取的信息优先探索，但仍需用户确认 |

## 持久化

达成所有共识后，直接持久化决策记录（不询问）。

保存路径：`.wok-grill/<YYYYMMDD>-<HHMM>-DECISIONS.md`

格式详见 [reference/decision-format.md](reference/decision-format.md)。

## 示例场景

- 新功能实现方案审视
- 架构设计压力测试
- 技术选型决策梳理
- 重构计划可行性验证
