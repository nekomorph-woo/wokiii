# 测试驱动开发

## 核心原则

**测试验证行为，而非实现细节。** 代码可以完全改变，测试不应受影响。

**GOOD tests**通过公共接口验证真实代码路径，描述系统做什么，而非怎么做。测试如规格说明书——"用户可用有效购物车结账"直接说明存在什么能力。此类测试经受内部重构，因为不关心内部结构。

**BAD tests**与实现耦合。它们 mock 内部协作者、测试私有方法、或通过外部手段验证（如直接查数据库而非使用接口）。警告信号：重构时代码行为未变但测试失败。若重命名内部函数导致测试失败，说明测试的是实现而非行为。

详见 [tests.md](reference/tests.md) 和 [mocking.md](reference/mocking.md)。

## 适用范围

**判断标准**：代码是否包含需要验证的业务规则或复杂逻辑。

| 判断 | 说明 |
|------|------|
| 需要 TDD | 有复杂逻辑、业务规则、边界条件需要验证 |
| 不需要 TDD | 只是传递数据、配置框架、调用第三方 |

**DO NOT** 询问用户哪些代码需要 TDD。若用户未明确指定，基于上述标准自行判断。

## 反模式：水平切片

**DO NOT** 先写完所有测试，再写所有实现。这是"水平切片"——把 RED 当作"写所有测试"，GREEN 当作"写所有代码"。

**正确做法**：通过 tracer bullet 进行垂直切片。一个测试 → 一个实现 → 重复。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 调用方式

无参数。任务通过对话上下文传入：

| 调用方 | 传入内容 |
|--------|---------|
| autopilot subagent | 从 plan 提取的接口、文件、验证标准（在 prompt 中） |
| autopilot 修复 CR | 从 review 提取的 finding、文件、修改方案（在 prompt 中） |
| 用户独立调用 | 对话中描述的任务 |

## 工作流程

### 1. 接收任务

从对话上下文或 subagent prompt 提取：

- **任务描述**：要实现什么（接口、功能、修复目标）
- **文件范围**：涉及哪些文件
- **验证标准**：如何判断完成

**autopilot subagent 模式**：任务已在 prompt 中明确描述，直接提取后跳到步骤 2。

**用户独立调用**：与用户交互确认：

- [ ] 与用户确认需要的接口变更
- [ ] 确认要测试的行为（优先级排序）
- [ ] 列出要测试的行为（非实现步骤）
- [ ] 获取用户确认

**信息不足时**：报告阻塞，输出缺少什么信息。不尝试自行补充，不读取任何管道文档。

### 2. Tracer Bullet

编写一个测试，验证系统的一件事：

```
RED:   编写第一个行为的测试 → 测试失败
GREEN: 编写使测试通过的最小代码 → 测试通过
```

### 3. 增量循环

对每个剩余行为：

```
RED:   编写下一个测试 → 失败
GREEN: 使测试通过的最小代码 → 通过
```

规则：

- 一次一个测试
- 只写使当前测试通过的最小代码
- 不要预判未来测试
- 保持测试聚焦于可观察的行为

### 4. 重构

所有测试通过后，寻找[重构候选](reference/refactoring.md)：

- [ ] 提取重复
- [ ] 深化模块（将复杂度移到简单接口之后）
- [ ] 在自然处应用 SOLID 原则
- [ ] 思考新代码揭示的现有代码问题
- [ ] 每步重构后运行测试

**RED 状态下不要重构。** 先回到 GREEN。

### 5. 完成报告

TDD 循环结束后，输出结构化报告：

```
✅ 完成: <任务摘要>
- 文件变更: <file list>
- 测试: <N> added, <M> passed
- 状态: GREEN
```

阻塞时：

```
⚠️ 阻塞: <任务摘要>
- 原因: <description>
- 需要决策: <what>
```

### 6. 会话模式萃取（可选，独立调用时）

独立编码会话结束后：

1. **回顾本次会话**，识别重复出现 3+ 次的实现模式
2. **应用严格质量门**（5 项准则全部满足）筛选候选
3. **检查已有规则**（读取 `.claude/rules/` 下所有 .md 文件），排除重复
4. 候选 < 2 条 → 跳过，不询问用户
5. 候选 ≥ 2 条 → 使用 AskUserQuestion 展示候选规则（最多 3 条），用户多选

## 每轮检查清单

```
[ ] 测试描述行为，而非实现
[ ] 测试仅使用公共接口
[ ] 测试能经受内部重构
[ ] 代码对当前测试是最小的
[ ] 没有添加投机性功能
```

## 约束

- **DO NOT** 读取 `_plan.md`、`_define.md`、`_review.md` 等管道文档
- **DO NOT** 修改任何管道文档（plan、review、autopilot log 等）
- **DO NOT** 进入 plan mode — 直接执行收到的任务
- impl 只知道：任务描述 + 文件范围 + 验证标准
