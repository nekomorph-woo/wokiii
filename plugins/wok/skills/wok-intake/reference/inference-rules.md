# 管道推理规则

按成果内容特征推理候选管道。输出 2-3 个候选，按置信度排序，第一个标 `(推荐)`。

## 信号 → 管道映射

### feat-（大功能）

**强信号**（命中 ≥ 2 个即高置信）：

- 多模块设计（≥ 2 个独立模块/组件）
- 完整功能描述（用户故事 + 验收标准）
- 涉及架构决策（分层、依赖关系、模块边界）
- 从零规划的新系统/子系统
- 跨多个 phase 的 roadmap（需要分阶段实现）

**弱信号**（命中后作为辅助证据）：

- 提到「系统」「平台」「框架」等大词
- 涉及多个用户角色
- 有显式的 milestone / 阶段划分

### feat-s-（小功能）

**强信号**：

- 单模块或 1-3 个紧密耦合模块
- 功能描述明确（用户故事 + 验收标准）
- 改动范围可控（不需要分阶段）
- 在已有系统上新增/增强功能

**弱信号**：

- 单一用户角色
- 简单的 UI 调整 / API 增强 / 配置项

### fix-（问题修复）

**强信号**：

- 报错信息 / 异常堆栈
- 根因分析（"为什么"类型讨论）
- 复现步骤 / 触发条件
- 修复方案（patch / 改动点）
- 用户描述包含「bug」「fix」「broken」「错误」「失败」「异常」

**弱信号**：

- 性能问题（慢、卡、超时）
- 行为不符合预期

### exp-（探索优化）

**强信号**：

- 代码探索 / 阅读理解
- 约束发现（架构约束、依赖关系、技术债）
- 模式总结（命名约定、代码风格、复用机会）
- 重构机会识别
- "理解 X 代码" / "看下 Y 怎么实现" 类讨论

**弱信号**：

- 提到「重构」「优化」「清理」
- 设计建议（但没到完整方案）

### 不推理 cr-（审查变更）

代码审查（`cr-`）的特征：

- 对已有代码变更做评估
- 输出问题清单 / Advisory
- "审查 X" / "review Y" 类讨论

**遇到 cr- 信号时**：不放入候选，提示用户：

```
当前成果属于代码审查场景，建议直接使用 /wok-code-review
（cr- 管道的产出 _review.md 不属于 plan 之前阶段，本 skill 不适配）
```

## feat- vs feat-s- 区分

两个都涉及"新功能"，关键区分点是**规模**：

| 维度 | feat-（大） | feat-s-（小） |
|------|-------------|---------------|
| 模块数 | ≥ 2 个独立模块 | 1-3 个紧密耦合 |
| 用户故事 | 多个，跨角色 | 1-2 个，单一角色 |
| 阶段 | 多 phase | 单 phase |
| 设计深度 | 完整架构 + 模块边界 + 接口契约 | 接口签名 + 实现要点 |
| roadmap | `_roadmap.md` 必需 | 无需 roadmap |

**模糊场景**：用户描述了功能但规模不清时，推荐 `feat-s-`（先小后大，避免过度规划）。

## 多候选排序规则

1. 按强信号命中数排序
2. 同分时按弱信号命中数排序
3. 仍同分时按推荐偏好：`feat-s-` > `fix-` > `exp-` > `feat-`（更小成本，先尝试轻量管道）

## 输出格式

```
候选管道（按置信度排序）：

1. [feat-s-] 小功能 (推荐)
   理由：单模块增强 + 明确用户故事 + 1-2 个验收标准
   命中信号：✓ 功能描述  ✓ 用户故事  ✓ 单模块

2. [feat-] 大功能
   理由：可能涉及多模块（待确认）
   命中信号：✓ 功能描述

3. [exp-] 探索优化
   理由：包含代码探索成分
   命中信号：✓ 模式总结（弱）
```

`AskUserQuestion` 中第 1 项标 `(推荐)`。
