# 执行步骤质量规范

每个执行步骤必须回答以下 4 个问题。

## 4 项必备要素

### 1. 文件范围

列出这一步涉及的文件：

- 新建文件：完整路径
- 修改文件：完整路径 + 修改区域描述
- **DO NOT** 写"视情况而定" — 必须明确

### 2. 接口清单

列出这一步要实现的接口：

- 从模块 `design.md` 中提取
- 标注哪些接口在这一步实现，哪些留到后续步骤

### 3. 验证标准

每个步骤必须有一个可通过的验证方式：

- 优先：自动化测试（TDD）
- 备选：编译检查、lint、手动验证脚本
- **DO NOT** 写"功能正常" — 必须可观测、可重复

### 4. 集成标注

标注这一步与其他模块的交叉点：

```
集成点: 调用 <模块B>.<接口X>()
前置: <模块B> 的 <接口X> 必须已实现
风险: <如果模块B的接口变更，这里需要同步修改>
```

## 遗漏风险识别

以下情况容易导致开发遗漏，需要特别标注：

| 风险信号 | 说明 | 应对 |
|----------|------|------|
| 一个步骤涉及 5+ 个文件 | 可能需要拆分 | 考虑拆为 2 个步骤 |
| 集成点无前置步骤 | 依赖模块还没做 | 调整执行顺序 |
| 验证标准不可自动化 | 依赖人工判断 | 考虑补充测试用例 |
| 步骤间有隐式依赖 | 步骤 B 需要步骤 A 的内部状态 | 显式标注依赖 |
